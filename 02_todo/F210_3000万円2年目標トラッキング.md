### Q1 案 a 修正版: DeviationSeverity Enum 新規導入

5 値 Enum を新設し、`severity_to_legacy_label()` で F211 既存 `severity:str` に
変換する。マッピングは:

```
AHEAD             → "ahead"     (新設、F211 既存 5 値になし)
ON_TRACK          → "ok"        (F211 既存)
MILD_DELAY        → "warning"   (F211 既存)
NEEDS_IMPROVEMENT → "severe"    (F211 既存)
RESET             → "critical"  (F211 既存)
```

`AHEAD → "ok"` を採用しない理由 (R-01-04 思想):
- R-01-05 では先行 (+5% 以上) も「無理に攻めない」が明示
- legacy `"ok"` にマッピングすると「順調 = 攻めて良い」誤読の余地
- 新ラベル `"ahead"` を立てて意味分離

新ラベル受理の根拠 (F211 関数本体での severity 参照は 2 箇所のみ):
- `goal/discipline.py:350` rationale 文字列 (デバッグ用)
- `goal/discipline.py:565` `EvaluationProposal.deviation_severity` 代入 (storage)
- いずれも `if`/`match` 分岐に使われない

### Q2 案 a: 境界値は `>=` で上側を取る (Phase 2 確認待ち)

```
+0.05 丁度 → AHEAD             (>= +0.05)
-0.05 丁度 → ON_TRACK          (>= -0.05)
-0.10 丁度 → MILD_DELAY        (>= -0.10)
-0.20 丁度 → NEEDS_IMPROVEMENT (>= -0.20)
それ以下    → RESET
```

要件書原文は「以上」「以下」「以内」「〜」混在で曖昧。Phase 2 で Fujiwara 確認。

`ValueError` fail-fast (`RuntimeError` ではなく `ValueError` で統一: 設定値

で実測ベース再評価予定 (実際の日本市場は年 245-248 営業日、祝日次第)。

## deviation_pct の意味 (F211 既存ドキュメントとの不整合解消)

F211 `goal/discipline.py:122` のインラインコメント `(current - target) / target、
負なら未達` は misleading。F211 既存テスト fixture
(`tests/goal/test_discipline.py:39-49`) は:

```
current_assets_yen=7_000_000.0
target_assets_yen=8_000_000.0
deviation_pct=-0.10                       ← 数式 (current-target)/target = -0.125 と不一致
elapsed_business_days=120
expected_progress_pct=0.5
actual_progress_pct=0.4
```

を持つ。すなわち `deviation_pct = actual_progress_pct - expected_progress_pct
= 0.4 - 0.5 = -0.10` (進捗ベース) が正しい解釈。F210 はこの整合解釈を採用し、
`calculate_deviation` 内で `actual - expected` を `deviation_pct` に格納する。
