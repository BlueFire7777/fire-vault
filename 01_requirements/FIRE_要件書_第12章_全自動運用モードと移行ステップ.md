# 第12章: 運用モードと移行ステップ(v3.3: 半自動主軸モデル)

## v3.3 での変更

v3.1/v3.2 は「最終目標 = 全自動運用」だったが、v3.3 では **最終目標 = Live Advisory(半自動主軸)** に変更する。Full Auto は将来のオプションとして位置づける。

## 最終目標の再定義

**v3.3 の最終運用形態:**

FIRE は **監督者付き自律運用システム**(第28章)として機能し、以下を実現する:

- FIRE が候補抽出・パターン照合・判断を自動で行う
- **発注指示は LINE 通知で人間(Fujiwara さん)に届く**
- Fujiwara さんが iSPEED / 楽天Web から手動で発注
- 約定データは Gmail API 経由 / LINE 手動報告で FIRE に戻る
- FIRE は約定結果を記録し、Pattern Store を更新する
- 人間は監督者として重要判断・承認・緊急対応を担う

これが v3.3 の「Stage 3 = Live Advisory」である。

## 半自動化する機能と、人間担当の機能

### 自動化する(FIRE が実行)

- 朝分析、候補抽出(1〜3銘柄厳選)
- Pattern Store 照合、再現性スコア算出
- 条件到達監視(利確・損切価格の到達検知)
- LINE 通知の送信(エントリー指示、利確指示、損切指示、緊急クローズアラート)
- 約定データの自動取込(M+2以降、Gmail API 経由)
- 建玉整合チェック(FIRE の想定と楽天証券実態の差分検知)
- 異常時の自動停止、Dashboard への状態表示

### 人間が実行(Fujiwara さん)

- 発注(エントリー・利確・損切・強制クローズ)- iSPEEDまたは楽天Webで手動実行
- 承認(Pattern 採用・凍結、Evaluation Agent 提案、モード変更)
- 緊急対応(想定外事象、システム障害、重大なDD発動時)
- 当日スケジュール共有(朝一言 or Google Calendar連携)

## 段階的検証と昇格ステップ(Stage飛ばし禁止)

| Stage | 名称 | 内容 |
|---|---|---|
| Stage 0 | Backtest | 過去データでルールベース検証 |
| Stage 1 | Replay Simulation | 過去特定日の時系列再生 |
| Stage 2 | Paper Live | リアルタイム相場で仮想売買、実発注なし |
| **Stage 3** | **Live Advisory(v3.3の実質ゴール)** | **リアルタイム相場で LINE 通知 → 人間発注。約定結果を FIRE に取込み** |
| Stage 4 | Full Auto(将来オプション) | エッジ確立後の自動発注拡張。優先度は低い |

**v3.3 での重要変更**: Stage 4 (Full Auto) は**初期目標から外す**。Stage 3 で十分な成績と運用安定性が得られれば、それが FIRE の完成形である。Full Auto への拡張は、エッジが M+3 以降で明確に確立し、かつ証券会社側の API 環境が整った場合のオプションとする。

## Stage 2 (Paper Live) → Stage 3 (Live Advisory) の昇格条件

### 必須条件

1. **サンプル数**: Paper Live を 20営業日以上 または 50トレード以上継続
   - ※ Mac mini の常時稼働を活かし、過去データ高速リプレイ検証とリアルタイム Paper Live の並列実行で、カレンダー日数を待たずに条件クリア可能
2. **主戦略の期待値**: プラス(勝率だけでなく平均利益・平均損失を含めて)
3. **最大ドローダウン**: 想定上限以内(総資産比 -5% を大きく超えない)
4. **Simulation Accuracy**: Optimistic Bias Score が許容上限以内
5. **LINE 通知基盤の動作確認**: 全通知部屋(エントリー/執行・損益/レポート/システム警告/**緊急ポジション整理**)が正常動作
6. **Fujiwara さんの受容体制確認**: 本業との両立が実運用で可能であることを確認

### 重要原則

- **「収益が出ていることだけを理由に本番移行してはならない」**
- 期待値、執行品質(仮想)、停止品質、シミュレーション現実性、**通知対応体制**が揃った場合にのみ、Evaluation Agent の提案を経てユーザー承認で昇格

## Stage 3 運用時のチェックリスト(日次)

Live Advisory 運用開始後、毎日以下をチェック:

- [ ] 朝レポートを 8:30 に受信したか
- [ ] 8:40 に実行候補通知を受信したか
- [ ] エントリー通知を受信したら、指示通り発注したか(or 見送り判断したか)
- [ ] 約定後、FIRE に約定結果が取り込まれたか(M+2以降自動、それ以前は手動LINE報告)
- [ ] 14:45 の建玉残アラートを確認したか
- [ ] 15:10 までに全建玉クローズしたか
- [ ] 引け後レビューを 16:00 に受信したか
- [ ] FIRE 想定建玉と楽天証券実態に差分がないか(整合チェック)

## 重複発注防止(v3.3版)

Stage 3 (Live Advisory) では、FIRE が自動発注しないため、v3.2 の「5層シグナルベース重複防御」は**不要**。代わりに以下の運用ルール:

1. **シグナル ID による通知ユニーク化**: 同一シグナル ID に対する LINE 通知は 1 回のみ
2. **Fujiwara さん側の「発注済み」フラグ管理**: LINE 通知に対して「発注済み」または「見送り」を返信
3. **FIRE 側の整合チェック**: 約定データ取込時に、同一シグナル ID の重複約定がないか確認
4. **Dashboard 表示**: 建玉数、発注残、約定残を可視化
