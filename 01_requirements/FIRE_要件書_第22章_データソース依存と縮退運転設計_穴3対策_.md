# 第22章: データソース依存と縮退運転設計(穴3対策)

第14章のデータソース優先順位を補完する詳細設計。FIREは多くの外部依存を持つため、「依存している」ことと「依存先が死んだら止まる」ことを分離する設計が必須。

## 穴3の本質

FIREの主な外部依存先は以下の通り。どれか1個が死んでも精度低下、複数が死ぬと事故になりうる。

| 系統 | 依存先 | 提供するもの |
|---|---|---|
| 楽天証券系(正本) | 楽天証券連携 | 余力・建玉・注文・約定 |
| 市場データ系 | 市場データAPI | 株価・出来高・ランキング・指数 |
| 材料系 | ニュースAPI / TDnet / IR | 好決算・上方修正・自社株買い・増配 |
| テクニカル系 | TradingView | チャート・相対強弱・テクニカル確認 |
| 通知/監督系 | LINE | 通知・停止・承認 |

**設計思想**: 主系統・補助系統・代替系統・停止条件を全部持つ。

## 危険パターン別の対策

### 危険1: 価格は取れるが楽天証券が死ぬ
- 対策: 新規発注停止 / 既存建玉の確認優先 / MCPで楽天画面確認に切替 / LINEシステム警告 / 自動運用はnotification onlyに落とす

### 危険2: 楽天は生きているが市場データが死ぬ(**最も危険**)
- 対策: TradingViewで補完できるか確認 → 安定しないなら新規停止 / 既存ポジの損切監視は楽天画面で補完 / 補完不能なら縮小orクローズ優先

### 危険3: ニュース/開示が死ぬ
- 対策: 新規の材料起因戦略の採用基準を厳格化 / 既存Pattern Libraryとテクニカル条件だけで候補出しするが件数を絞る / 決算シーズンは特にconservativeモード

### 危険4: TradingViewが死ぬ
- 対策: 自前テクニカル計算エンジンを持つ / 楽天画面や市場データで代替

### 危険5: LINEが死ぬ
- 対策: ダッシュボードへログ蓄積 / ローカルアラートログ保存 / Semi Autoでは運用停止 / Full Autoでは通知不能を危険状態として扱うかをモードで分ける

## 縮退運転モード(5段階)

| モード | 発動条件 | 運用内容 |
|---|---|---|
| 通常モード | 全ソース生存 | 通常運用 |
| 限定運用モード | 一部ソース欠損 | 候補数を減らす・ロットを減らす・守備化 |
| 通知専用モード | 執行ソースが危険 | 候補監視と通知だけ |
| 保有管理専用モード | 新規執行不可 | 新規は止めて建玉管理だけ継続 |
| 緊急停止モード | 致命的障害 | 全新規停止・必要なら全クローズ |

## 設計原則(5原則)

1. **正本は1つに固定**: 余力/建玉/注文/約定 = 楽天証券のみ
2. **価格・監視は複線化**: API + TradingView + 楽天画面
3. **材料系は遅れたら守備化**: 精度低下時は攻めない
4. **通知不能時はモード依存で止める**: 承認必要なら停止
5. **「止まる」ことも成功**: データソース異常時に止まるのは負けではなく健全

## Dashboardで監視するデータソース健全性指標

| カテゴリ | 指標 |
|---|---|
| 可用率 | 楽天接続可用率 / 市場データ取得成功率 / ニュース取得成功率 / 開示取得成功率 / TradingView MCP可用率 / LINE送信成功率 |
| 遅延 | 価格更新遅延 / 注文応答遅延 / 通知遅延 |
| モード遷移 | 通常→限定運用になった回数 / 通知専用モード回数 / 緊急停止回数 / 原因別ランキング |

## 運用思想の明文化

FIRE は、データソース欠損時にも無理に通常運用を続けない。情報の鮮度・正確性・執行可能性が担保できない場合は、運用レベルを段階的に下げ、必要に応じて新規停止または緊急停止へ移行する。
