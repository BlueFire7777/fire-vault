# 第14章: データソース優先順位と縮退運転(v3.3)

## v3.3 での変更

v3.1 では「楽天証券 = 余力・建玉・注文・約定の正本」とし、RSS 経由で自動取得する前提だった。v3.3 では発注を人間が行うため、**楽天証券からのデータ取得は「約定通知メール経由」となる**。

## データソース優先順位(v3.3改訂版)

| 優先度 | ソース | 用途 | 取得方式 |
|---|---|---|---|
| **1位(正本)** | **楽天証券(約定通知メール経由)** | 建玉、約定、実現損益 | M=0〜M+1: LINE で Fujiwara さん手動報告 / M+2 以降: Gmail API 自動取込 |
| 2位 | 市場データAPI | 日経225、TOPIX、為替、原油、個別株価 | 公式API |
| 3位 | TradingView(MCP) | チャート、ライン、テクニカル確認、相対強弱 | TradingView MCP |
| 4位 | TDnet / IR API | 適時開示、決算短信 | 公式API |
| 5位 | ニュースAPI | 市況、個別材料 | 公式API |
| 6位(補助) | Google Calendar | Fujiwara さんの当日スケジュール認識 | Google Calendar API |

## 重要な位置づけ変更

### 余力の取り扱い

v3.1 では「余力は楽天証券から自動取得、発注時に参照」だった。v3.3 では発注が人間主体なので、**余力は Fujiwara さんが発注時に自分で確認する**。FIRE は発注指示時にロット計算するが、余力の最終確認は Fujiwara さんが実行画面で実施。

**補助**: FIRE は想定余力を日次で管理(前日残 + 日中の約定差分)。Fujiwara さんが朝に「余力 XXX 万円」と LINE 入力すれば、FIRE の想定余力が実態と一致する。

### 建玉の正本

建玉の正本は**楽天証券**。FIRE は以下を管理:

- **FIRE想定建玉**: シグナル発行 → Fujiwara 発注指示 → 約定報告 の積み上げ
- **楽天証券実態建玉**: 約定メール(自動取込) or 朝のFujiwara手入力から算出
- **差分チェック**: 両者に差分があれば Dashboard で警告、Fujiwara さんに通知

## 縮退モード(v3.3 版)

### OpenClaw が生きていて、外部データソースが死んだ場合

- **楽天証券約定メール取込失敗**: Fujiwara さんに LINE で「約定手動報告してください」通知
- **市場データ死亡**: 新規エントリー停止、既存建玉は Fujiwara さんに状況確認依頼
- **TDnet/IR 死亡**: 材料起因戦略の候補採用基準厳格化、守備化
- **TradingView死亡**: 自前テクニカル計算で補完、またはチャート確認を Fujiwara さんに依頼
- **Gmail API 死亡**: M+1 までの運用(LINE 手動報告)に一時的にフォールバック

### OpenClaw が死んだ場合(v3.3 で新設)

第22章(データソース縮退)と連動。詳細は同章参照。
