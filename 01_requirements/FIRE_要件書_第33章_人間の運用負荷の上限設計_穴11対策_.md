# 第33章: 人間の運用負荷の上限設計(穴11対策)

## 穴11の本質

FIREは自動化を目指しているが、**設計を盛りすぎると「提案が多い・Dashboardが重い・承認が多い・LINE通知が多い」になってユーザーが疲れる**。

## 運用負荷の上限目安

| 区分 | 上限 |
|---|---|
| 日次確認時間 | 朝10分 + 引け後10分 = 合計20分以内 |
| 場中の介入 | 原則ゼロ(異常通知が来た時のみ) |
| 承認対象の件数 | 1日最大3件まで(それ以上は翌日以降に繰り越し) |
| LINE通知の件数 | 緊急・重要のみ。情報通知は朝レポート・引け後レポートに集約 |
| 週次レビュー時間 | 30分以内 |
| 月次レビュー時間 | 1時間以内 |

## 通知の階層化

| 階層 | 内容 | 通知タイミング |
|---|---|---|
| 緊急(即時) | 緊急停止・重大障害・損失上限到達 | 即時LINE |
| 重要(要確認) | 承認待ち・昇格条件到達・Evaluation Agent重要提案 | 即時LINE |
| 情報(まとめ) | 候補通知・執行結果・引け後サマリー | 朝レポート・引け後レポートに集約 |
| 参考(Dashboard) | パターン成績・詳細分析・週次月次レポート | Dashboard閲覧時のみ |

## 承認が必要なものの最小化

承認対象は以下に限定し、それ以外は自動実行する。

**承認必須**: full auto移行 / Pattern採用・凍結・DEATH NOTE / Evaluation Agent提案採用 / 大きなリスク設定変更 / 緊急停止解除

**自動実行(承認不要)**: 通常の候補抽出・執行・利確・損切・強制クローズ / 朝レポート・引け後レポート送信 / Paper liveの仮想売買

## Dashboard初期KPIの絞り込み

初期に表示するKPIは最小限に絞り、後から拡張する。

**初期必須(10項目以内)**: 今日の損益 / 累積損益 / 目標乖離率 / 今日の実行候補 / 主戦略の勝率 / 建玉状況 / システム稼働状態 / 承認待ち件数 / 直近の停止理由 / Evaluation Agent提案数

**後から追加**: パターン別詳細成績 / 特徴量重要度 / Simulation精度 / 週次・月次詳細分析
