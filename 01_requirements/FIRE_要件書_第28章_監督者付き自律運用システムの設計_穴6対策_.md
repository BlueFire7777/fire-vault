# 第28章: 監督者付き自律運用システムの設計(穴6対策)

## 穴6の本質

FIREは全自動を目指すが、**全自動化と無監督化を混同すること**が穴6。この2つは別物。

| 区分 | 内容 |
|---|---|
| 全自動化(目指す) | 自動監視・自動選別・自動執行・自動停止・自動評価 |
| 無監督化(目指さない) | 人間は何も見ない・通知も見ない・異常時も任せる・長期間ノーチェック |

## FIREの理想運用形態

FIREは、人間を相場監視から解放するが、**人間を運用責任から切り離すものではない**。FIREは**無監督ロボットではなく、監督者付きの自律運用システム**として設計する。

| 要素 | 内容 |
|---|---|
| 通常時 | 自動 |
| 異常時 | 通知 |
| 定期的 | 人が監督 |
| 重要変更 | 承認制 |
| 停止権限 | 必ず人間が持つ |

## 運用レベルの定義(5段階)

| レベル | 名称 | 内容 | 目標 |
|---|---|---|---|
| レベル1 | 通知支援 | FIREが候補提案。人間が発注・管理 | — |
| レベル2 | 仮想自律 | FIREが全部動く(paper live)。人間は監督だけ | — |
| レベル3 | 半自動 | FIREが判断。発注は承認後。人間は重要時だけ操作 | — |
| レベル4 | 自律運用 | FIREが自動執行。人間は監督者。緊急停止と承認変更を担当 | **ここまでが到達目標** |
| レベル5 | 完全無監督 | 人間がほぼ見ない | 目指さない |

## 人間の監督頻度(現実的な目安)

| 頻度 | 内容 |
|---|---|
| 日次(朝) | 稼働状態確認 |
| 日次(引け後) | 結果をざっと確認 |
| 場中 | 通常は張り付かない。異常通知が来た時だけ見る |
| 週次 | Evaluation Agentの提案を見る / パターンの生死を確認 |
| 月次 | 目標進捗 / リスク量 / パターン構成 / DEATH NOTE / REHABを見直す |

## 人間が必ず関与するもの vs 通常は関与しないもの

| 区分 | 内容 |
|---|---|
| 必ず関与 | 初期モード変更 / full auto移行承認 / Pattern採用・凍結・DEATH NOTE / Evaluation Agent提案採用 / 緊急停止解除 / 大きなリスク設定変更 |
| 通常は関与しない | 条件監視 / paper live / 仮想売買 / 自動通知 / 主戦略の通常執行 / 利確・損切・強制クローズ |

## LINEとDashboardの役割分担

| ツール | 役割 |
|---|---|
| LINE(即時監督UI) | 即時通知 / 異常通知 / 緊急停止 / 承認・却下 / 状態確認コマンド |
| Dashboard(運用監督UI) | 詳細確認 / 運用全体の監督 / 評価提案レビュー / パターン管理 / 進捗確認 |

## 完全放置を防ぐ設計ルール(4つ)

| ルール | 内容 |
|---|---|
| Heartbeatチェック | 3営業日以上Dashboard / LINE確認なしの場合、監督注意通知を送信 |
| 重要変更は必ず承認 | パターン変更 / リスク変更 / モード変更 / 本番移行はすべて承認必須 |
| 異常復帰は手動 | 緊急停止したら、勝手にfull autoに戻らない |
| 長期ノータッチ防止 | 一定期間レビュー未実施なら、Full Auto継続に警告を出す |

## 穴1〜6の総括

| 穴 | 問題 | 対策 |
|---|---|---|
| 穴1: エッジが弱い | エッジの根拠が曖昧 | 再現性優位で定義(第26章) |
| 穴2: 約定品質 | 執行品質が悪い | 執行品質ゲート(第17章) |
| 穴3: データソース依存 | 依存先が死ぬと事故 | 縮退運転(第22章) |
| 穴4: 広げすぎ | 最初の焦点がぼやける | 主戦略集中(第24章) |
| 穴5: 目標で攻めすぎ | 目標圧力で規律が崩れる | 目標と攻撃性を切り離す(第27章) |
| 穴6: 完全放置 | 無監督化で事故が起きる | 監督者付き自律運用(第28章) |
