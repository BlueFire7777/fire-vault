### 確定: 仮説 A — features 未生成

**根拠 (ソースレベル)**:

`simulation/paper_live/tick.py` L139〜193 の `extract_candidates()` 抜粋:

```python
def extract_candidates(ctx: TickContext) -> list[PaperLiveResult]:
    ...
    # features.dt 規約 (F055): "{date}T09:00:00+09:00" で書き込まれている
    date_str = ctx.as_of_dt.strftime("%Y-%m-%d")
    feature_dt = f"{date_str}T09:00:00+09:00"

    from patterns.reproducibility import ReproducibilityEngine
    engine = ReproducibilityEngine(db_path=ctx.db_path)

    candidates: list[PaperLiveResult] = []
    for symbol in symbols:                              # 4,452 銘柄全件
        if not _has_features(symbol, feature_dt, ctx.db_path):
            continue                                     # ← ここで全銘柄即 skip
        ...
        result = engine.evaluate(symbol=symbol, dt=feature_dt, top_n=10)
        ...
```

**features 不在の銘柄は ReproducibilityEngine.evaluate() に到達せず即 continue**。

3 run の挙動 = 4,452 銘柄スキャン × 120 日 × 各 tick で `_has_features=False` 全部 skip → events=0。

## 3. 修正案 (STEP 2、Fujiwara 承認待ち)

### 案 1: extract_features に batch モード追加 + Core500 scope (推奨)

**変更内容**:
- `scripts/jobs/extract_features.py`:
  - `--all-symbols` フラグ追加 (market_listings から取得)
  - `--symbols 1234,5678,...` カンマ区切り対応 (--code を非 required 化)
  - `--scope core30 / core100 / core500` プリセット (TOPIX 流動性上位)
  - `--parallel N` で multiprocessing 並列化 (default = CPU 数 ≒ 12)
  - `COLLECTOR_STATUS["material"]["ready"] = True` 修正 (F101 完了反映)
- `scripts/jobs/backfill_features.py` 新規 (薄いラッパー、`--scope core500 --date-from --date-to`)

**工数**: 4 時間 (実装 2.5h + 単体テスト 1h + 動作確認 0.5h)
**処理時間**: M4 Pro 12 並列で **Core500 銘柄 × 120 営業日 ≈ 1〜1.5 時間**
**features テーブル膨張**: 約 36 万行 (= 500 × 120 × 6 feature_key、現状 87 行から +400 倍だが許容範囲)
**副作用**: 案 1 完了後に Run a 再実行 → events>0 を確認 → Run b/c も再実行で **約 5 時間**で全 Chain 再走可能

**メリット**:
- TOPIX Core500 は楽天証券で売買可能な主要銘柄をほぼカバー
- 流動性重視の FIRE 戦略 (R-17-07 出来高ガード) と整合
- DB サイズ膨張も合理的範囲

**デメリット**:
- 上場直後の小型株 (流動性高化候補) はカバー外 → patterns 候補発見の機会損失
- `--scope` プリセット定義 (TOPIX Core500 リスト) が J-Quants にあるか確認必要

### 案 2: 全 4,452 銘柄一括 backfill (現実的でない)

**処理時間**: M4 Pro 12 並列で **4,452 × 120 × 5 秒 ÷ 12 ≈ 6 時間**
**features 膨張**: 約 320 万行
**メリット**: 完全カバレッジ
**デメリット**: 6 時間 + DB 1 GB 級膨張、しかも実機 Stage 3 で 4,452 銘柄監視は非現実的 (R-33 運用負荷上限抵触)

→ **不採用**

### 推奨

**案 1** で進める。実装 4 時間 + Core500 backfill 1.5 時間 + 全 Chain 再走 5 時間 = **約 11 時間で events>0 達成見込み**。今夜 (2026-05-04 夜) にスタートすれば翌朝 5 時頃までに完了。

ただし以下は Fujiwara 判断:

1. **scope プリセット定義**: TOPIX Core500 を直接的に取れる API がない場合、market_listings から流動性ベース (出来高 × 株価上位) で 500 銘柄抽出する代替実装が必要。これは追加 1〜2 時間。
2. **announcements 補完 (副因 B 解消)**: 仮説 A 解決後、announcements を直近 6 ヶ月遡及取得するタスク (F101 拡張) が必要。これは案 1 完了後の別タスクで OK。
