## 3. ケース 4-B 確定 (events_total < 50)

### 3-1. 確定根拠

- Step 3 実測 events_total = 0 (1 day batch_replay)
- paper_live_results / paper_live_positions すべて空
- candidate 抽出経路自体が動作していない or score 0.5787 < 0.65 で execute
  されず candidate event が記録されない

### 3-2. 仮説 (本部判断材料)

仮説 1: extract_features 経路で build_snapshot_from_db が呼ばれていない
- Phase 2-B Vault handover §4-2 + §8 で「extract_features 経路への collector
  連携: Phase 2-B では snapshot 取得経路のみ確立、collect() が呼ばれる経路は
  既存 (将来 build_snapshot_from_db 呼出の wiring が必要なら別 task)」と
  明記済
- TOPIX 58 行は index_data に蓄積されたが、Run a の tick で
  build_snapshot_from_db 経由で MarketSnapshot が組まれていない可能性
- → regime_fit は default 0.5 のまま、Phase 2-B 効果ゼロ

仮説 2: extract_features 経路で snapshot 構築されているが score 0.65 未満
- Phase 1 推定 score 寄与: regime_fit 0.5 → 0.7-0.9 で +0.02〜+0.04
- 0.5787 + 0.04 = 0.6187 < 0.65、execute されず
- candidate event は出るが execute せず paper_live_results に記録されない

仮説 3: 別の構造的原因
- patterns 4,708 件 (Phase 2-A 着手前 DB 確認) が candidate 抽出経路で
  動作不全 (F273 Phase 2-A の Layer 1 seeder 結果が dt フィルタで除外?)
- pattern_research_agent が F275 完了時点で events>0 達成しなかった事実と
  整合 (F275 complete report line 121-122)

### 3-3. F271 §6-7 適用 (線形外挿楽観視回避の実測検証完了)

Phase 1 推定 (events 50-100 件、現実シナリオ) は **悲観シナリオ域** (1-5 件)
or それ以下 (0 件) であることが Step 3 実測で確定。Mac mini Phase 1
handover §C で示した「悲観シナリオ: positions seeding でも score 上昇限定的、
5-20 件」よりさらに悪い結果。

→ §6-7 (線形外挿楽観視回避) の適用が再度必要。Phase 2-C 微調整も実測検証
必須。
