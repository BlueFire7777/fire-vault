## 2. §B 全文: seeding script CLI 仕様 (G-2 承認)

### 2-1. ファイル

新規: `~/fire/scripts/jobs/run_backtest_seeding.py`

### 2-2. 実装方針

既存 F230 batch_replay (`simulation/paper_live/batch_replay.py`) の薄いラッパー
+ cron 親和性のための CLI 整形。`BatchReplayRunner` を直接呼ぶ、新規 batch
logic は不要。

### 2-3. CLI 引数

```
.venv/bin/python -m scripts.jobs.run_backtest_seeding \
    [--days-back 60]            (default、b3 推奨値、--start-date と排他)
    [--start-date 2026-02-05 --end-date 2026-05-02]  (期間明示指定)
    [--patterns ID1,ID2,...]    (default: active 系全件 = approved_active +
                                 watchlist)
    [--fill-model strict]       (default、F052 標準、F276 todo §A 推奨)
    [--interval-min 5]          (default、tick 間隔)
    [--skip-lunch]              (default ON、11:30-12:30 除外)
    [--dry-run]                 (営業日リスト + 計画のみ表示、実行なし)
    [--json]                    (JSON 出力)
```

### 2-4. exit code 規約 (F300 hardening 整合)

| code | 意味 | 条件 |
|---|---|---|
| 0 | success | events_total ≥ 50 達成 |
| 1 | error | DB error / fetch error / batch_replay 内部例外 (silent return 0 解消) |
| 2 | partial | events_total > 0 だが < 50、batch 完走済 |

### 2-5. 内部呼出フロー

```
run_backtest_seeding (CLI)
  ↓ argparse
  ↓ dotenv load (~/fire/.env)
  ↓ 営業日リスト計算 (--days-back or --start-date/--end-date)
  ↓ --dry-run 分岐 (営業日リスト + 計画 print → exit 0)
  ↓
BatchReplayRunner.batch_replay(
    days_back=N,
    patterns=...,
    fill_model="strict",
    interval_min=5,
    skip_lunch=True,
)
  ↓ 内部で paper_live_positions / paper_live_results に INSERT
  ↓ result: BatchReplayResult (n_events / n_positions_opened / closed)
  ↓
exit_code = 0 if result.n_events >= 50 else 2
            (例外発生時は 1 + sys.stderr に JSON)
```

### 2-6. cache invalidation 整合 (F271 v1.2 §3.6.10)

paper_live_positions writer-side (内部 BatchReplayRunner) で
`reset_trade_stats_cache()` 既呼出 (F275 で確立済)、本 wrapper では追加呼出不要。

### 2-7. テスト要件

- `tests/scripts/test_run_backtest_seeding.py` (新規):
  - --dry-run: 営業日リスト出力 + INSERT なし
  - --json: JSON 出力検証
  - exit code 0/1/2 各分岐
  - 小規模実行 (--days-back 5、active 1 pattern) で INSERT 件数検証

## 3. §E 全文: index_data schema + idempotent migration 仕様 (G-5 承認)

### 3-1. ファイル

新規: `~/fire/scripts/setup/migrate_index_data.py`

### 3-2. schema (新規 DB / 既存 DB 統一)

```sql
CREATE TABLE IF NOT EXISTS index_data (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    date         TEXT NOT NULL,           -- YYYY-MM-DD
    symbol       TEXT NOT NULL,           -- 'TOPIX' / 'N225' (MVP)
    open         REAL,                    -- 始値
    high         REAL,                    -- 高値
    low          REAL,                    -- 安値
    close        REAL NOT NULL,           -- 終値 (regime 算出に必須)
    change_pct   REAL,                    -- 前日比 % (-1.0〜+1.0、collector 用)
    volume       REAL,                    -- 出来高 (任意)
    created_at   TEXT NOT NULL,           -- ISO timestamp
    UNIQUE (date, symbol)                 -- 冪等性確保
);
CREATE INDEX IF NOT EXISTS idx_index_data_date
    ON index_data(date);
CREATE INDEX IF NOT EXISTS idx_index_data_symbol_date
    ON index_data(symbol, date);
```

### 3-3. idempotent 確認 (F271 v1.2 §3.6.5、F300-race 同型)

1. `CREATE TABLE IF NOT EXISTS` で既存テーブル時は noop
2. `UNIQUE (date, symbol)` 制約 + `INSERT OR IGNORE` で重複行は再挿入されない
3. 同 migration を 2 回連続実行 + dump 行数差分ゼロ確認
4. test: `tests/scripts/test_migrate_index_data.py` で 2 回実行検証 +
   UNIQUE 制約発動確認

### 3-4. 既存資産同型

`scripts/setup/migrate_paper_live_notifications.py` (F300-race) の
ALTER TABLE ADD COLUMN + IF NOT EXISTS 経路を踏襲。新規テーブル CREATE
のみで ALTER は不要。

### 3-5. index_fetcher 仕様 (G-4 承認)

新規: `~/fire/market_data/index_fetcher.py`

- F100 aiohttp 同型流用 (market_data/historical.py + client.py)
- 対象指数: TOPIX + 日経 225 (MVP、G-4 確定)
- J-Quants API 経由:
  - TOPIX: `/markets/topix` または `/indices/quotes`
  - 日経 225: `/indices/quotes` (code=N225 等、J-Quants 仕様確認後確定)
- DB 書込: `INSERT OR IGNORE INTO index_data ...`
- exit code 0/1/2 (F300 hardening 整合)
- timeout 明示 (`aiohttp.ClientTimeout`)、ClientSession `async with` で必ずクローズ

### 3-6. features/regime.py 連携

### 3-7. cron 登録 (Fujiwara 手動、本 commit 範囲外)

```

    --cron "0 16 * * 1-5" --tz Asia/Tokyo --no-deliver \
    --message "python -m market_data.index_fetcher --json"
```

平日 16:00 (引け後) に index_fetcher を起動、当日終値を index_data に蓄積。

### 4-3. G-3: 期待 events 数 50-100 件、§6-7 実測検証必須

楽観/現実/悲観シナリオ:
- 楽観: 60 営業日 × 各 pattern × symbol で 0.05 件/tick = 750 件
- 現実 (★採用): positions seeding 効果でスコア 0.5787 → 0.7 上昇、
  SCORE_THRESHOLD_EXECUTE 0.65 突破で 50-100 件
- 悲観: positions seeding でも score 上昇限定的、5-20 件

§6-7 適用: 上記は線形外挿、Phase 2-A 完了後に **1 day batch_replay 実測** で
予測確度を検証必須。実測値が悲観シナリオ域なら Phase 2-C で
SCORE_THRESHOLD_EXECUTE 微調整 (0.65 → 0.55) を本部 Q3 で承認するパスが必要。

### 4-4. G-4: index_fetcher 対象指数 TOPIX + 日経 225 (MVP)

根拠:

  market_regime のみ
- `classify_market_regime()` ヒューリスティクス (`features/regime.py:98`) が
  この 2 指数で完結
- 米株指数 (S&P500/NASDAQ) / グロース 250 / JPX 日経 400 は将来拡張余地

### 4-6. G-6: 案 P シリアル採用 (A 完走 → B 着手)

根拠:
- A の batch_replay は 60 営業日 × Core500 = 数時間級の重い処理、並走中の B が
  J-Quants API レート制限を圧迫すると A の features 抽出も影響受ける可能性
- Mac mini 24 時間稼働で同日内 2 タスク完了 (A: 4-6 時間 + B: 1-2 時間 +
  Run a 再走: 30 分) は十分可能、シリアルでも実工数 1 日内
- 時間帯分離 (案 Q) は人為ミス (時間ずれ起動) リスクあり、シリアルの方が確実
- 完全並走 (案 R) は API レート制限リスク + cache 競合可能性

シリアル運用ガード:
- A 完走確認 4 項目 PASS で B 着手:
  1. exit code 0 確認
  2. paper_live_positions の events_total 実測値 ≥ 50
  3. Codex pre-commit CRITICAL ゼロ
  4. commit + push 完了
- 1 項目でも NG なら B 着手見送り、本部判断仰ぐ
