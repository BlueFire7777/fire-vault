## なぜこの順序か

### 1. Stage 3 移行を遅らせない

買い側はもう 955 PASS まで来ていて、あと Mac mini 実行 + 環境設定 + 最終承認で Stage 3 開始です。ここで空売りに着手すると:

- F111 (Daytrade Selection) に direction 軸 → 既存テスト 955件への影響広範囲
- F115 (Trade Decision) ロット計算式に空売り分岐 → F132/F133/F140 連鎖改修
- Pattern Store schema migration (direction カラム) → F027/F028/F031/F032 全件書き直し

最低でも +200〜300 PASS、2-3週間級。**Stage 3 がそのぶん遅れる**。

### 2. 空売りは初期 Paper Live のみ = 緊急性が低い

仕様書方針上、M=0〜M+1 は Paper Live のみ。M+2 = Stage 3 開始から最低2ヶ月後。**M+1 終盤までに Pattern Store schema と空売りロジックが入っていれば間に合う**。

### 3. 買い側の実運用ログを見てから空売り設計したほうが質が高い

執行品質、スリッページ、LINE運用、iSPEED 発注フロー の **実データ**が取れる。空売りの設計判断 (価格乖離上限、損切スリッページ、踏み上げ実測値、iSPEED での信用売建発注UI) は買い側の実運用ログを見てから決めたほうが、推測ベース実装より質が高い。

F230 / F100 historical / F241 のときと同様、**指示書は推測ベース、実装側が実コード/実 API を見て修正**のパターンを継続できる。

## Phase 別タスク内訳

### Phase 1: 基盤実装 (M=0 終盤〜M+1 序盤)

**A. データソース実装**
- A-1. J-Quants `/listed/info` の貸借区分取得 (Standard プラン項目)
- A-2. J-Quants `/markets/short_selling` 業種別空売り比率
- A-3. J-Quants `/markets/short_selling_positions` 空売り残高報告
- A-4. JPX 信用取引規制銘柄スクレイピング (日次)
- A-5. JPX 空売り規制トリガー抵触銘柄スクレイピング (日次)
- A-6. 楽天証券一般信用売り在庫取得 (Phase 1 は手動 → CSV 取込)

**B. Pattern Store schema migration**
- B-1. patterns テーブルに direction カラム追加 (デフォルト "long")
- B-2. patterns テーブルに setup_type 拡張 (S1〜S5 enum 追加)
- B-3. patterns テーブルに空売り固有カラム追加 (borrow_status / regulation / inventory / reverse_interest_risk / regime / sector_flow / index_trend)
- B-4. F027/F028/F031/F032 を direction 別に分離
- B-5. F031 類似度検索を direction フィルタで限定

**C. 空売りスコア・ロジック実装**
- C-1. 空売りスコア計算 (R-10-06 〜 R-10-10、100点満点)
- C-2. 絶対見送り条件のスコア前段ゲート
- C-3. 候補抽出フィルタ (貸借/売禁/在庫/トリガー/出来高/主役テーマ等)
- C-4. S1 悪材料初動ロジック (F101 を direction 反転で流用)
- C-5. S4 VWAP割れロジック (純テクニカル判定)

**D. リスク管理実装**
- D-1. 空売りリスク係数 (R-05-15、買いの半分)
- D-2. ロット計算式 (STOP > ENTRY、100株丸め)
- D-3. 空売り専用停止条件 (2連敗 / 日次-0.6% / 踏み上げ損切 / 指数急反発)
- D-4. 14:00 以降新規ショート禁止ガード

**E. LINE 通知テンプレ**
- E-1. 空売りエントリー通知テンプレ (R-06-21、※既存 R-06-14 と衝突回避でリナンバリング後)
- E-2. 空売り返済通知テンプレ (R-06-22)
- E-3. 空売り損切通知テンプレ
- E-4. 信用売建/返済買 用語統一バリデーション (誤発注防止) (R-06-23)

### Phase 2: Paper Live 検証 (M+1 終盤)

**F. Paper Live 稼働**
- F-1. F230 の direction 拡張 (空売り Paper Live)
- F-2. 空売り tick.py 中身実装 (STOP/TP 反転)
- F-3. 10〜20 営業日リプレイ
- F-4. setup_type 別の期待値・勝率・最大DD 集計

**G. 昇格判定**
- G-1. F241 類似の空売り版 (--check-short-live-advisory)
- G-2. setup_type 別の昇格判定 (S1/S4 個別に判定)
- G-3. Fujiwara さん最終承認

### Phase 3: 100株実弾 (M+2)

- 期待値+ かつ執行可能性確認できた setup_type のみ実弾化
- 100株固定で開始
- 買い側と同じ LINE 通知 + iSPEED 発注フロー

### Phase 4: 本格運用 (M+3 以降)

- 基準 (期待値・最大DD・勝率・平均RR・執行品質) クリアで本格運用
- 買いの半分のリスクで運用 (R-05-15)
- S2/S3/S5 は F104 完成後に追加

## リスク・依存関係

| リスク | 対応 |
|---|---|
| Phase 1 着手判定が早すぎて Stage 3 安定運用を妨げる | 買い側の M=0 で問題が出ていないことを確認してから Phase 1 着手 |
| Pattern Store schema migration で既存テストが大量に壊れる | 既存パターンに direction="long" デフォルト値を強制適用、F027/F028 のテストに direction 別分岐を追加 |
| 一般信用売り在庫データが楽天証券から取得できない | Phase 1 は手動 CSV 取込で代替、Phase 2 以降で自動化検討 |
| S2/S3/S5 のデータソースが見つからない | F104 (指数) + F102 (ニュース API) の完成を待つ。MVP は S1+S4 のみで成立 |
| 空売り Paper Live で期待値プラスが出ない | 撤退判定する。空売りは補助レーン位置付けなので、無理に実装しない判断もあり得る |

## 撤退基準

以下の場合、空売り対応を撤退または凍結する判断を Fujiwara さん最終承認で行う:

- Phase 2 (Paper Live) で 20 営業日リプレイしても期待値マイナス
- 踏み上げ損切りが 5 回以上発生
- 楽天証券の一般信用売り在庫データが安定取得できず、運用に支障
- 買い側の運用に集中したほうが ROI が高いと判断された場合

撤退時は v3.5 で「空売り対応は v3.4 で凍結、買いに集中」として要件書を更新する。
