## 2. 本部レビュー結果

### 2-1. 論点 1: tick 失敗通知の送信先

**判断**: 承認。

- 主送信先は `Room.SYSTEM`
- 建玉残あり時のみ `Room.EMERGENCY` に追加送信
- メッセージ本文は SYSTEM / EMERGENCY 共通テンプレートを採用

共通テンプレート採用理由:
- 実装が単純で分岐漏れリスクが低い
- 運用上の緊急度差は送信先の部屋名で十分に区別できる
- 本文を分ける必要が出た場合は別タスクで拡張可能

### 2-2. 論点 2: reproducibility.py 記述修正

**判断**: 承認。

Phase 2 commit 2 (B) で F277 スタブの記述を実態に合わせて修正する。

修正前に本部へ差分レビューを依頼すること。

確定した実態:
- `patterns/reproducibility.py` に signature 関数は元々存在しない
- 「F275 で削除した signature 関数を復活」は誤記
- 修正対象は `patterns/reproducibility.py` line 62-96 の F277 言及 TODO コメント削除のみ
- signature SQL 経路の復活は `patterns/similarity.py` の TTL default 変更で達成する

### 2-3. 論点 3: _has_features の扱い

**判断**: 承認。

`_has_features` は F275 で外部 API 互換性維持のため残すと明示されている。
`extract_candidates` 経路では `_fetch_symbols_with_features` による一括取得へ移行済みで、
`_has_features` は未参照である。

raise 化対象は以下 4 箇所に縮減する。

| 関数 / 経路 | 方針 |
|---|---|
| `_fetch_all_symbols` | `return []` ではなく raise |
| `_fetch_symbols_with_features` | `return set()` ではなく raise |
| `_fetch_ohlc` | `return None` ではなく raise。個別銘柄エラーは上位で counter |
| `engine.evaluate` (`extract_candidates` 内) | `continue` ではなく raise。個別銘柄エラーは上位で counter |

`_has_features` 定義箇所には、F271 §6-3 忘却リスク回避のため以下の趣旨のコメントを追加する。

```python
# F277 (2026-05-04): extract_candidates 経路では
# _fetch_symbols_with_features 経由 (一括取得) で未参照。
# 本関数は外部 API 互換性のための残存。
# 将来的に外部呼び出し元がなくなった時点で削除候補。
# F271 §6-3 忘却リスク回避のため削除判断は要件書側の確認を経て別タスク化すること。
```

### 2-4. 論点 4: 個別銘柄エラーの扱い

**判断**: シンプル案を承認。

- 構造的エラーは raise
- 個別銘柄エラーは continue + counter
- counter が `tick_failure_threshold_per_run` を超えたら raise
- counter は `runner.py:tick()` メソッド内のローカル変数とし、run_id 単位で初期化する
- プロセス全体累積にはしない

#### 閾値 50 の根拠

`tick_failure_threshold_per_run = 50` は、総銘柄数約 4,000 銘柄のうち
**1.25% を超える連続エラーは構造的問題と判断する**ための閾値である。

計算:

```text
50 / 4000 = 0.0125 = 1.25%
```

FIRE の paper_live tick は多数銘柄を順次評価するため、数件の個別銘柄データ欠損や
計算例外はあり得る。一方、50 件を超える連続失敗は DB ロック、スキーマ不整合、
特徴量テーブル破損、評価エンジン側の共通不具合など、個別銘柄ではなく構造的問題の
可能性が高い。

この閾値はハードコードせず、`~/fire/config/` 配下の設定として
`tick_failure_threshold_per_run = 50` を定義する。

## 3. sqlite3 例外階層調査結果

### 3-1. 調査環境

- Python: 3.9.6
- SQLite: 3.51.0
- `sqlite3` module version: 2.6.0
- 調査方法: 対象ランタイムで `sqlite3` 例外クラスの `mro()` / `__bases__` を確認

### 3-2. 例外階層

| 例外 | 継承 |
|---|---|
| `sqlite3.Error` | `Exception` |
| `sqlite3.DatabaseError` | `sqlite3.Error` |
| `sqlite3.OperationalError` | `sqlite3.DatabaseError` |
| `sqlite3.IntegrityError` | `sqlite3.DatabaseError` |
| `sqlite3.DataError` | `sqlite3.DatabaseError` |
| `sqlite3.InternalError` | `sqlite3.DatabaseError` |
| `sqlite3.ProgrammingError` | `sqlite3.DatabaseError` |
| `sqlite3.NotSupportedError` | `sqlite3.DatabaseError` |
| `sqlite3.InterfaceError` | `sqlite3.Error` |
| `sqlite3.Warning` | `Exception` |

### 3-3. F277 での判定境界

F277 では `sqlite3.DatabaseError` 配下を構造的エラーとして扱い、raise 対象にする。

対象:
- `sqlite3.OperationalError`: database is locked、no such table、schema 変更、接続・I/O 系
- `sqlite3.IntegrityError`: 制約違反。paper_live 状態記録としてはデータ整合性問題
- `sqlite3.DataError`: DB データ処理エラー
- `sqlite3.InternalError`: SQLite 内部エラー
- `sqlite3.ProgrammingError`: SQL バインドミス、closed connection など実装不整合
- `sqlite3.NotSupportedError`: API 利用不整合

`sqlite3.InterfaceError` は `DatabaseError` 配下ではないが、DB API インターフェース層の
不整合であり、発生した場合は構造的エラーとして扱う余地がある。Phase 2 実装では
`sqlite3.DatabaseError` を主境界とし、`sqlite3.InterfaceError` を含める必要があるかは
既存コードの catch 粒度に合わせて commit 1 着手時に最終確認する。

個別銘柄エラーとして continue + counter 対象にするのは、DB 例外ではない銘柄固有の
計算例外・データ値例外に限定する。

## 5. 段階性ガード 3 ポイント計測

F271 §6-7 線形外挿楽観視回避、および §6-8 計測対象取り違え回避のため、
計測経路は `ReproducibilityEngine.evaluate()` 経由に固定する。

`SimilarityEngine` 単独計測は補助情報のみとし、完了判定には使わない。

| タイミング | 計測対象 | 追加観点 |
|---|---|---|
| A 単独完了時 | per-call 計測。期待値は F275 比 ±0 ms | failure 経路の所要時間。SYSTEM / EMERGENCY 送信込み |
| A+B 完了時 | per-call 計測 | TP / SL 後の次 tick で `_get_trade_stats_cached` の SQL 再実行コストが想定通りか確認 |
| A+B+C 完了時 | per-call 計測 + Run a 1340 ticks 実走 | C による per-call +1〜2 ms 想定、Run a 35 分前後予測 |

受入基準:
- per-call < 5 ms 維持
- Run a 1340 ticks < 60 分維持
- Stage 3 前提の安全性課題が構造的に解消されている

## 7. Codex pre-commit 運用

F277 では `--no-verify` を使わない。

F274 / F275 で 2 回連続 `--no-verify` が使われたため、F277 では最低 1 回は
pre-commit を厳格通過させる必要がある。構造的問題で例外承認が必要な場合は、
commit 前に本部チャットへ事前協議する。

Codex 自身が pre-commit hook のレビュー実行者であるため、Codex が開発者として
commit する場合は自己レビュー構造になる。通常の pre-commit を通せばよいが、
CRITICAL 指摘が出た場合は commit 前に解消または本部協議する。
