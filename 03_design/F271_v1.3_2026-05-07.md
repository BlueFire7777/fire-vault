## 2. §6 アンチパターン集 (v1.2 → v1.3、+11 項目)

### §6-7 系 (線形外挿楽観視回避、3 項目追加)

#### §6-7-bis: Vault 未確定マーカー事前解消

  **要旨**: 各タスク着手前に Vault 内「要確認」「要検討」「要追加調査」
  マーカーを grep + 解消、Vault 未確定状態のまま実装着手しない。

  **事例**: F276 Phase 2-B 実装時に F276_phase2_design line 161
  「J-Quants 仕様確認後確定」を確認スキップ → 試行運用で endpoint 不在
  発覚。

  **回避ルール**: タスク着手前に Vault grep 必須、未確定マーカー全数解消。

#### §6-7-quaternary: 閾値依存性構造不変実証

  **要旨**: 閾値微調整 (例: 0.55 → 0.60) で events 数は 1000 倍変動するが
  勝率/期待値の構造は不変、score モデル基盤の問題を閾値で解決不可。

  **事例**: F276 Phase 4-α (env=0.55、events 103,312、勝率 6.6%) →
  Phase 4-β (env=0.60、events 39,420、勝率 6.5%) で実証。

  **回避ルール**: 閾値変更で構造改善を期待しない、score モデル/features/
  patterns の本質的見直しが必要な場面を識別。

#### §6-7-quinary: シミュレーション数値出力時の物理的可能性 + 実運用前提整合確認

  **要旨**: シミュレーション数値出力時、物理的可能性 (実取引で達成可能な
  範囲) + 実運用前提 (個人投資家の取引数、ロット数、信用利用) との整合を
  確認せず、楽観的数値を提示する本部側ミス。

  **事例**: 本セッション本部が「期待値 +5,000 円 → 月利益 +6M 円」と試算、
  Fujiwara 指摘で Backtest 数値 (4,377 件/day) と実取引数 (個人 1 日 5-10
  件) の混同が発覚。

  **回避ルール**: シミュレーション数値出力時は実取引前提明示、Backtest
  数値と実取引数を区別、本部記憶ベース推測を Vault 確定値で裏付け。

### §6-22 系 (本部側 Vault 参照漏れ、8 項目追加)

#### §6-22-a: チャット応答 Vault 化済誤認識

  チャット応答内容を Vault 化済と誤認識して指示。チャット応答テキストは
  Vault に明示的に commit しない限り正本でない。

#### §6-22-b: Phase 1 連鎖参照漏れ

  Phase 1 inventory 観点で関連設計記録を引用したが、過去 Phase 結果まで
  連鎖参照せず。例: F276 Phase 1 で F040 design 引用するも F273 Phase 2
  結果を未参照。

#### §6-22-c: wiring 別 task 起票要否確認漏れ

  「将来 wiring 別 task」と Vault に書きつつ、別 task 起票要否を確認せず
  Phase を進める。例: F276 Phase 2-B「将来 wiring 別 task」を Phase 4
  着手前に未起票確認。

#### §6-22-d: 本部記憶ベース順序推測

  本部記憶で「F235 後着手」「Run b/c 先行」等の順序推測、Vault 確定済
  「並走可」「過去データ判定」と乖離。

#### §6-22-最重大事例: 本部判断時の Vault 突合不在で金銭的損害発生

  **事例**: F276 案 a (J-Quants Premium 加入推奨) で Fujiwara に約 13,200
  円/月の sunk cost 発生 (Premium で N225 取得不可確定済を本部 Vault 突合
  せず推奨)。F276_indices_spec_2026-05-05.md §3「N225 取得不可は Standard/
  Premium 区分の問題ではない」確定済を本部判断時に未参照。

  **回避ルール**: コスト発生案件 (金銭的判断/契約変更/物理的に戻らない判断)
  は Mac mini 経由 Vault 突合 + 公式 spec WebFetch + Fujiwara 確認の三重
  確認必須化 (HQ-Lean v1.1 ルール 11)。

#### §6-22-bis: Fujiwara 提供文書の核心定義文要素分解必須

  **事例**: 本セッションで Fujiwara 提供文書「各レーンの期待値・地合い適性・
  劣化状況・執行品質・DD を評価しながら、実弾通常/縮小/Paper/監視/停止を
  切り替える」核心定義文を本部が要素分解せず MVP 12 項目のみで整理。
  「執行品質」「地合い適性」を独立評価軸として MVP に含めない本部側ミス、
  Fujiwara 指摘で発覚。

  **回避ルール**: Fujiwara 提供文書の核心定義文を要素分解 (評価軸/切替
  モード/性格等)、各要素が MVP に含まれているか逐一検証、文書末尾の MVP
  リストだけで採用範囲確定しない。

#### §6-22-tertiary: 新方針/大規模戦略変更時の MVP 範囲過小評価バイアス

  **事例**: 本セッションで本部が F281 段階 2 MVP を「Lane A1 単独構造」と
  整理 → Fujiwara 指摘で「Lane B-E + G/H も段階 2 で開始可能」と判明、
  本部側で範囲過小評価していた。

  **回避ルール**: 新方針採用時、Fujiwara 戦略観 (b 確信 = FIRE が exploit
  可能) を踏まえた包括的 MVP 範囲検討必須、既存 features / 既存実装で対応
  可能な範囲を本部側で勝手に「新規実装必要」と過大評価しない。

#### §6-22-quaternary: 新方針の核心思想「レーン単位独立稼働」を「同時開始」と本部誤解

  **事例**: 本セッションで本部が Stage 3 開始想定 6/15-7/20 (6 レーン完成
  待ち) と整理 → Fujiwara 指摘で「OK になったレーンから本番運用開始」に
  修正、Stage 3 開始 5/22 → 5/14 頃に短縮可能と判明。

  **回避ルール**: 新方針の核心思想 (独立稼働 vs 同時開始) を Fujiwara 提供
  文書で確認、本部側で「同時開始」と推測断定せず、レーン単位の独立稼働
  経路を採用。

#### §6-22-quinary: Vault 確定済 Run a/b/c による R-32-01 過去データ判定経路を本部内で逆転理解

  **事例**: 本セッションで本部が「Lane A1 単独で 20 営業日 + 50 トレード
  蓄積に時間がかかる」と整理 → Fujiwara 指摘で「過去のトレードで回せば
  すぐ」と判明、Vault 確定済の Run a/b/c 経路 (R-32-01 ※注記) を本部内で
  正しく反映できていなかった。

  **回避ルール**: R-32-01 等の判定基準が「過去データ判定 vs 実取引判定」の
  いずれかを Vault 確定値で確認、本部記憶ベースで「実取引で蓄積」と推測
  断定しない。

#### §6-22-senary: 本部側保守バイアス過剰により Stage 3 開始が遅延方向に偏る傾向

  **事例**: 本セッションで本部が新方針追加・6 レーン同時誤解・本質課題
  対応必須化等で Stage 3 開始を 5/8 → 6/15-7/20 まで段階遅延 → Fujiwara
  指摘「どんどん運用開始から遠ざかっていく」で発覚。

  **回避ルール**: Stage 3 開始時期見積時、本部側保守バイアス (責任回避の
  ため安全側に振る傾向) を自覚、Fujiwara 戦略方針 (損益直結優先、贅肉
  削減) と整合する最速経路を提示。

### §6-23 系 (本部記憶 vs Vault 一次資料、4 項目)

  Mac mini 報告内容を本部側で Vault 突合せず鵜呑みにする。

  着手指示書「F100 aiohttp 同型流用」、実態 requests 同期実装)。

  本部記憶で進行順序を推測、Vault 確定値との照合をスキップ。

  **要旨**: 同一番号系列内で F276-d-A-fix / F276-Phase4-recovery 等の起票

  **要旨**: 本部判断は Mac mini 経由 Vault 突合で裏付け、本部記憶ベースの

  **要旨**: 本部記憶推測の進行順序を Vault で裏付け、推測ベース指示禁止。

  **事例**: 「Run b/c 先行 → F235 後着手」推測 vs Vault「F278/F276/F235 並走」

#### ルール 11: コスト発生案件三重確認

  **要旨**: 金銭的判断 / 契約変更 / 物理的に戻らない判断は Mac mini 経由
  Vault 突合 + 公式 spec WebFetch + Fujiwara 確認の三重確認必須。

  **事例**: F276 案 a Premium 加入で sunk cost 約 13,200 円発生
  (§6-22-最重大事例)。

#### ルール 12: Mac mini 並行指示上限管理

  **要旨**: 未完了指示がある状態で新規指示を追加発行しない、未完了 →
  完了報告 → 本部応答 → 次指示の順序維持。

  **適用**: 本部指示書発行前に Mac mini の完了状態確認、未完了の指示が
  ある場合は完了待機後に新指示発行。

#### ルール 13 (v1.4 新設): 設計記録作成タスクの完了基準に既存 Phase 連鎖参照確認 (grep ベース) を必須化

  **要旨**: 設計記録 (~/fire-vault/03_design/F*.md) を新規作成 / 改訂
  する際、関連既存 Phase との schema / 実装整合確認 (grep + DB schema
  確認) を完了基準に含める。

  **適用**:
  - 設計記録作成タスク着手時: 関連キーワード (lane / schema / pattern /
    migration 等) で Vault grep + ~/fire 実コード grep
  - 完了基準: 既存 Phase の言及が設計記録に明示されている、または「既存
    実装なし」を grep 結果で確認済み
  - 設計記録末尾の関連リンクに既存 Phase task / commit を引用

  **事例**: §6-22-septenary。F281 設計記録 v1.0 で F200 言及不在 →
  F281-Phase1-bis で改訂 v1.1 + 本ルール新設。

#### ルール 14 (v1.5 新設): e2e / smoke / 統合テストの完了基準にテスト用注入データの cleanup を必須化

  **要旨**: e2e / smoke / 統合テスト等で patterns / positions / その他の
  運用 DB に注入するテストデータは、テスト完了基準に「archive 化 / 削除 /
  status 降格」処理を必須として含める。

  **適用**:
  - テスト設計時に「クリーンアップ手順」を明示 (todo 完了基準 + 設計記録)
  - テスト後の DB 状態が運用 DB と区別可能な状態 (例: status='archive') に
    なっていることを完了基準とする
  - Run / 評価系タスクで DB 状態を参照する前に「test 由来レコード」の影響
    範囲を確認

  **事例**: §6-22-octonary。F058 e2e smoke test (2026-05-01) で
  `--inject-test-pattern` により approved_active 1 件注入、F058 完了時に
  archive 化処理を明示せず、5 タスク連続で残置 → 2026-05-07 Run a 着手前に
  発覚。

#### ルール 15 (v1.5 新設): log.md milestone 内の未解決事項を後続タスクで grep + 報告必須化

  **要旨**: log.md milestone 内で「要再評価」「未検証」「要確認」「F266 で
  再評価必要」等のキーワードで問題提起された未解決事項は、後続タスクの
  着手時に grep で必ず参照、未解消なら本部に明示報告。

  **適用**:
  - タスク着手前に `grep "要再評価\|未検証\|要確認\|再評価必要\|未対応" log.md`
    実行
  - 該当タスクと関連する未解決事項 (前 Phase 結果 / DB 状態 / spec 確定等)
    を本部に列挙報告
  - 未解消のまま着手する場合、本部判断 (許容 / 解消優先 / scope 拡大) を
    取得

  **事例**: §6-22-octonary。log.md line 609-614「F266 Stage 3 ゲートで
  再評価必要」と line 1721-1727「target_patterns 既定値が status='approved_active'
  の 1 件のみを抽出する仕様 (推定、未検証)」が F274 / F275 / F276 / F281 の
  4 タスクで参照されず、Run a 着手前まで放置。
