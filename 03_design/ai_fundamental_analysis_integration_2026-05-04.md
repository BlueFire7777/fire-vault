# AI ファンダメンタル分析の FIRE への組み込み構想

**作成日**: 2026-05-04
**位置づけ**: 中期 (Stage 3 → Stage 4 移行検討時) の機能拡張構想
**前提**: Stage 3 = Live Advisory が完成、実取引データが蓄積された後の改善余地

---

## 1. 背景

FIRE は再現性ベース (テクニカル・パターンマッチ) のアプローチで、以下の特性を持つ。

- 4000 銘柄並列スキャン、機械的・定量的・再現可能
- バックテスト + Run a 1340 ticks 等の検証フレームあり
- 個別銘柄の文脈 (業界トレンド・経営戦略・決算カタリスト) は捉えにくい

X (Twitter) で観察される sunaiper80 系の AI 決算翌日株価予想は、対照的なアプローチ。

- 1 銘柄 1 イベント深掘り、定性分析中心
- LLM (Claude / GPT 等) で業績・事業内容・市場環境を分析
- 採点 + レンジ予想 + 上がる条件を出力
- 構造化テンプレ (採点 / 結論 / 上がる条件 / レンジ / コメント) で量産
- ただし過去予想の的中率トラッキングは公開されておらず検証困難

両アプローチには構造的な強み弱みがあり、市場局面と時間軸で使い分けが必要。
本構想は両者の補完関係を活用したハイブリッド設計を記録する。

---

## 2. 両アプローチの比較

### FIRE (再現性・テクニカル) が強い場面

- 流動性高い銘柄 (大型・主力)
- イベントなしの平常時の値動き
- 短期 (デイトレ〜数日)
- 多銘柄並列で機会捕捉数が多い戦略

### AI ファンダメンタル分析が強い場面

- 構造変化のあるカタリスト (決算 / 業績修正 / M&A)
- 個別銘柄の文脈が決定的な場面
- 中長期 (数週間〜数ヶ月)
- 流動性の低い小型・新興株

### 構造的な特性比較

| 観点 | FIRE | AI ファンダ |
|---|---|---|
| 単位 | 4000 銘柄並列 | 1 銘柄深掘り |
| 入力 | 価格・出来高・テクニカル | 業績・事業・市場 |
| 判断主体 | 完全機械処理 | LLM + 人間 |
| 時間軸 | デイトレ〜数日 | 翌日〜中長期 |
| 検証性 | バックテスト可能 | 過去予想の的中率次第 |
| スケーラビリティ | 高 | 低 |
| 市場 AI 普及への耐性 | 低 (パターン消失リスク) | 高 (情報処理速度差で残る) |

## 3. ハイブリッド構想 (中期実装案)

### 基本方針

FIRE が広く銘柄をスキャンして候補を出し、AI ファンダ分析が「業績文脈の構造化採点」を
加える。両者の合意が取れた銘柄で勝負する。

### 観察ベースの設計 (sunaiper80 系の分析から)

X (Twitter) で観察される AI 決算翌日株価予想の出力構造を分析した結果、以下の要素が
有効と判断。

#### 取り入れる要素

##### 3.1 6 軸構造化スコアリング

| 軸 | 評価内容 | 入力データ | 配点 |
|---|---|---|---|
| 1. 業績サプライズ | 会社予想・市場予想からの乖離率 | F101 XBRL 業績数値 | 0-20 |
| 2. 来期ガイダンス | 増益率 / 最高益更新の有無 | F101 適時開示 | 0-20 |
| 3. 過熱度 | 直近株価上昇率 / RSI / ボリンジャー | FIRE features (既存) | 0-15 |
| 4. 需給 | 信用倍率 / 出来高変化 / 機関の動向 | F100 J-Quants V2 (一部要拡張) | 0-15 |
| 5. セクタートレンド | 同業他社の業績 / 株価動向 | 新規データソース必要 | 0-15 |
| 6. 安定性 | カタリストの分散 (単一材料 vs 複数ドライバー) | 上記 5 軸の分散 | 0-15 |

合計 0-100 点、閾値 (例: 80 点) 超で候補化。

##### 3.2 揃い度スコア (派生指標)

5 軸 (1〜5) のスコアの分散から自動計算する派生スコア。

```
揃い度 = 1 - (5軸スコアの標準偏差 / 平均)
```

- 5 軸全部が高得点 → 揃い度高い → 「安定して上がりそう」
- 1 軸だけ突出 → 揃い度低い → 「一発材料頼み」

これにより sunaiper80 の「本命 vs 対抗」の差別化を主観的順位付けではなく数値で表現。

##### 3.3 レンジ予想の出力

| シナリオ | 算出方法 | FIRE への連動 |
|---|---|---|
| 楽観 | 過去類似パターンの上位 25% | TP 候補 |
| 中央値 | 過去類似パターンの中央値 | 期待リターン |
| 悲観 | 過去類似パターンの下位 25% | SL 候補 |

不確実性を明示する点で重要。FIRE の TP/SL 自動設定と整合させる。

##### 3.4 「上がる条件」「下がる条件」の明示化

LLM に「この銘柄が上がるための具体的な条件を 4 つ挙げよ」と指示。これにより。

- 条件が不成立になった瞬間に手じまい判断ができる
- 後から検証可能 (「来期ガイダンス強気」が外れたから下がった、等)
- F271 §6-2「動いた / 機能した混同」回避 (条件を明示することで偽成功を排除)

##### 3.5 時間軸明記

出力に予想時間軸を必ず含める。

- next_day: 翌日予想 (sunaiper80 系の標準)
- one_week: 1 週間以内
- one_month: 1 ヶ月以内

時間軸別に的中率を分けて記録、FIRE の TP/SL 設定と整合 (デイトレなら next_day を採用)。

##### 3.6 イベント駆動モード + 常時モード

| モード | 起動契機 | 対象銘柄数 | API コスト |
|---|---|---|---|
| イベント駆動 | 決算発表前後 ±3 日 | 100〜200 銘柄/日 | 低 |
| 常時 | 毎日全銘柄 | 4000 銘柄/日 | 高 |

両方サポート。Stage 4 着手時はイベント駆動モードから始め、効果検証後に常時モード拡張。

##### 3.7 構造化 JSON 出力

```json
{
  "ticker": "1930",
  "ticker_name": "北陸電気工事",
  "post_date": "2026-04-28",
  "score_breakdown": {
    "earnings_surprise": 18,
    "guidance": 17,
    "overheating": 12,
    "supply_demand": 14,
    "sector_trend": 13,
    "stability": 10
  },
  "total_score": 84,
  "alignment_score": 0.85,
  "scenario": {
    "optimistic_pct": 8.0,
    "central_pct": 5.5,
    "pessimistic_pct": -2.0
  },
  "time_horizon": "next_day",
  "rise_conditions": [
    "来期ガイダンス強気維持",
    "電設セクター好調継続",
    "決算翌日の出来高増",
    "需給良好"
  ],
  "fall_conditions": [
    "市場全体の地合い悪化",
    "セクター内競合の悪材料",
    "事前期待の織り込み済み",
    "為替急変"
  ],
  "evidence": [
    {"type": "earnings", "value": "経常利益+18.2%、会社予想+30%上振れ"},
    {"type": "guidance", "value": "来期+13.7%増、最高益見通し"},
    {"type": "sector", "value": "電設セクター好調"}
  ],
  "prompt_version": "fundamental_scorer_v1.0"
}
```

#### 取り入れない要素

| 要素 | 理由 |
|---|---|
| 主観的順位付け (本命 / 対抗) | FIRE は絶対スコアと閾値で判断、揃い度スコアで代替 |
| 投稿候補の事前フィルタ | FIRE は全銘柄を平等にスコアリング (低スコアも記録) |
| 曖昧表現 (期待が高い等) | 数値根拠の明示を必須 (例: 業績期待 = 会社予想比 +30% 以上) |

### 実装イメージ

#### ステップ 1: 既存 F101 (TDnet/IR) の活用

F101 で取得済の以下を AI ファンダ分析の入力にする。

- 適時開示 (decisions / earnings / IR releases)
- 決算発表日・業績修正タイミング
- XBRL 業績数値 (Phase 3)

#### ステップ 2: LLM スコアリングエージェント

新規エージェント `agents/fundamental_scorer.py`。

- 入力: 銘柄コード + 6 軸入力データ + 時間軸指定
- 処理: 構造化プロンプトで Claude / GPT に 6 軸採点 + 揃い度 + レンジ予想 + 上下条件を出力
- 出力: 構造化 JSON (上記フォーマット)
- 検証: 過去予想と実績の照合 (的中率トラッキング)
- プロンプト: バージョン管理 (`agents/prompt_templates/fundamental_scorer_v*.md`)

#### ステップ 3: FIRE スコアとの統合

```
final_score = w1 * pattern_score + w2 * fundamental_score
TP = max(pattern_TP, fundamental_optimistic_TP)
SL = min(pattern_SL, fundamental_pessimistic_SL)
```

重み (w1, w2) はバックテストで最適化。

#### ステップ 4: 決算カタリスト特化モード

- 決算発表前後 (発表日 ±3 日): fundamental_score の重み上昇
- 平常時: pattern_score 重視

#### ステップ 5: セクタートレンド機能 (新規データソース)

5 軸目「セクタートレンド」は現在の F100/F101 では取得できない。
追加データソース候補。

- 業界別の業績集計 (同セクター企業の決算結果集計)
- セクター ETF の値動き
- 業界ニュース (新規スクレイピング機能)

これは Stage 4 着手時に F101 拡張または新規 F タスクとして起票検討。
最小実装 (案 C2) では 3 軸 (業績 / ガイダンス / 過熱度) のみで Phase 1 を構成し、
需給 / セクター / 安定性は Phase 2 で拡張する。

### 段階的実装方針 (案 C2)

| フェーズ | 軸 | 工数 | 必要データ |
|---|---|---|---|
| Phase 1 (最小実装) | 業績 / ガイダンス / 過熱度 | 5 日 | F100/F101 既存 |
| Phase 2 (拡張) | + 需給 | 2 日 | F100 拡張 |
| Phase 3 (完成) | + セクター / 安定性 | 3 日 | 新規データソース |

Phase 1 で効果検証、効果あれば Phase 2/3 拡張。効果なければ撤退。

## 4. 観察フェーズ (Stage 3 開始までに実施)

### 目的

実装前に sunaiper80 等の AI 決算翌日株価予想の的中率を独自検証する。
「AI ファンダ分析が本当に効くのか」を実装前に判断する材料を取得。

### 実施方法

CSV 形式で観察ログを蓄積。場所: `~/fire-vault/04_observation/sunaiper80_log.csv`

#### CSV 列定義

```
post_date, ticker, ticker_name, score, scenario, optimistic_pct, central_pct,
pessimistic_pct, conditions, actual_next_day_pct, accuracy_judgment, notes
```

| 列名 | 内容 |
|---|---|
| post_date | 投稿日 (YYYY-MM-DD) |
| ticker | 銘柄コード |
| ticker_name | 銘柄名 |
| score | AI 採点 (0-100) |
| scenario | 投稿テンプレ種別 (next_day_top / individual_deep / etc) |
| optimistic_pct | 楽観予想 (%) |
| central_pct | 中央値予想 (%) |
| pessimistic_pct | 悲観予想 (%) |
| conditions | 上がる条件 (改行区切り) |
| actual_next_day_pct | 翌日実績 (%) |
| accuracy_judgment | 評価 (hit / miss / partial) |
| notes | 備考 |

### 観察期間と判断基準

- 観察期間: Stage 3 開始後、運用稼働 1〜3 ヶ月の中で実施判断
  (Stage 3 開始時点で改めて観察フェーズの開始可否を Fujiwara 判断、
   FIRE 単独運用との並行観察か、観察フェーズ自体を省略して実装時に
   過去ツイート遡及検証するかを選択)
- データ量目標: 50 件以上の予想を記録
- 判断基準:
  - 80 点以上の的中率が 60% 超 → AI ファンダ統合の価値あり、Stage 4 で着手判断
  - 80 点以上の的中率が 50% 程度 → 効果限定的、参考情報止まり
  - 80 点以上の的中率が 50% 未満 → 統合不要、別アプローチ検討

## 6. リスクと注意点

### リスク 1: LLM 予測精度の未検証

現在の LLM の予測精度はコインフリップを少し上回る程度の可能性あり。組み込み前に
過去予想の的中率を観察フェーズで独自検証する必要あり。

### リスク 2: 後付け説明 (post-hoc rationalization) リスク

LLM は「上がる理由」「下がる理由」を流暢に書けるが、書けることと当たることは別。
F271 §6-2 (動いた / 機能した混同) と同型のリスク。

### リスク 3: AI 普及によるテクニカル優位性の消失

逆に FIRE 側のリスクとして、AI 取引の普及でパターン優位性が消える可能性。
中長期で観察必要。

### リスク 4: API コストの爆発

4000 銘柄全部に LLM を投げると現実的でない。イベント駆動モードで決算前後 ±3 日に
対象を絞る、または常時モードでも候補絞り込み後に LLM 投げる設計が必須。

### リスク 5: プロンプトインジェクション

外部から取得する適時開示テキストに悪意ある文字列が混入した場合の防御。
LLM への入力サニタイズ必須。

### リスク 6: LLM の主観的バイアス

LLM は「本命 / 対抗」のような主観的順位付けを得意とするが、これは過去予想の的中率と
相関しない可能性。FIRE 統合時は数値ベースの絶対スコアのみ採用、主観的階層化は排除。

### リスク 7: 投稿バイアス (生存者バイアス)

外部の AI 予想アカウント (sunaiper80 等) は、当たった予想だけ目立つ生存者バイアス
あり。参考にする際は過去予想の全件追跡が必要 (当たった + 外れた両方)。観察フェーズで
全件追跡 (sunaiper80 のタイムラインを定期的に確認、後出し編集を排除)。

### リスク 8: 6 軸の重み付けの最適化困難

6 軸採点は理論的に妥当だが、実際の重み付け (どの軸を重視すべきか) はバックテストで
試行錯誤が必要。Stage 4 で実装時、初期重みは均等から始め、データ蓄積後に
最適化。

### リスク 9: AI 補助の過信

AI ファンダ分析は FIRE の補助情報であり、単独で投資判断しない。LLM の主観的バイアス
を過信せず、必ず FIRE の検証フレームを主軸として残す。
