## 2. 3 案の前提分岐分析

### 案 A: DB schema + PRE-INSERT (`INSERT OR IGNORE` 原子的予約)

**修正方針**:
1. `paper_live_notifications` の **PRIMARY KEY (position_id, event_type)
   制約は既存** (UNIQUE は事実上既に確保済)
2. 送信**前**に `INSERT OR IGNORE INTO paper_live_notifications` を実行
3. `cursor.rowcount > 0` (= 自分が初めて予約した) ならば送信担当
4. `cursor.rowcount == 0` (= 別 process が既に予約済) ならば skip
5. 送信失敗時の cleanup: `DELETE WHERE position_id=? AND event_type=?`
   (rollback)

**現実装からの差分** (`agents/monitoring_alert.py:97-168`):

```
変更前:
  _is_already_notified (SELECT) → format → send → _record_notified (INSERT)
                                  ↑ ここで race window 発生

変更後:
  _try_reserve (INSERT OR IGNORE + rowcount 判定) → if reserved: format
    → send → if failed: _release (DELETE)
```

**Pros**:
- 並行起動の有無に依らず安全 (DB 制約で構造的に重複排除)
- 全 deployment 環境で一貫した safety
- schema 変更は最小 (status カラム追加は不要、PK 制約だけで十分)
- F300 哲学 (構造的解消、運用設定に依存しない) と整合
- INSERT OR IGNORE + rowcount は SQLite で原子的に動作 (WAL モード OK)

**Cons**:
- `MonitoringAlertAgent.process_events` の経路改修 (現実装の 4 段階を
  再構成、test 影響あり)
- 送信失敗時の DELETE rollback 設計 (LINE API failure → DB cleanup)
- INSERT 直後 process crash → orphan 行 (永久 skip) リスク → 状態カラム
  (pending/sent) で TTL 失効が必要なら schema 拡張、ただし MVP では
  「crash = 通知漏れ容認」と割り切る選択肢もあり

**修正範囲**:
- `agents/monitoring_alert.py` (process_events / _record_notified の
  経路改修)
- `scripts/wrapper/price_monitor.py` (LEFT JOIN は維持、二重防御として)
- `tests/agents/test_monitoring_alert.py` (race scenario 追加)
- schema migration: 不要 (PK 制約は既存)、ただし状態カラム追加なら
  `scripts/setup/migrate_paper_live_notifications.py` 新規

**実装工数**: 0.7-1.0 日 (本部見積と一致)

### 案 B: filelock

**修正方針**:
1. `pip install filelock` (requirements.txt に追加)
2. `scripts/wrapper/price_monitor.py` 起動時に `filelock.FileLock` 取得
3. lock 不取得 (既に他 process 実行中) → 即終了 (exit 0 or 専用 exit code)

**Pros**:
- schema 変更不要
- wrapper レベルで排他化、コード変更も限定的
- filelock ライブラリは pip 1 行で導入可

**Cons**:
- filelock 依存追加 (F265 で python-dotenv 追加した経緯と方針整合性
  検討必要、依存最小化方針と矛盾)
- 異常終了時 stale lock 検出機構 (filelock は OS レベルで通常解放
  されるが、SIGKILL / kernel panic 時は stale 残存)
- 排他粒度が wrapper 全体 = price_monitor が長時間処理中、別の
  もう 1 個も完全待機 (現状は問題なくても将来拡張時の並列処理性能
  ボトルネック)
- 「並行起動なし」前提に倒す = F300 の構造的解消哲学に反する
  (case を扱わない設計、Phase 1 見落としと同型構造)

**修正範囲**:
- `scripts/wrapper/price_monitor.py` (filelock context 追加)
- `requirements.txt` (filelock pin)
- `tests/scripts/test_price_monitor_wrapper.py` (新規、lock 取得 race
  シナリオ)

**実装工数**: 0.3-0.5 日

### 案 D: 運用側責務 (OpenClaw cron 設定)

**修正方針**:
1. OpenClaw cron に「並行起動防止」設定で fire_intraday_monitor 登録
2. 手動起動禁止を運用ルールで明示 (Fujiwara 認知負荷)
3. `agent_registration_guide.md` に運用注意 + plist 整合性チェック手順
   を明示

**Pros**:
- コード変更ゼロ
- 最小修正 (docs のみ)
- OpenClaw cron が単一スケジューラなら、5 分毎設定で overlap 発生の
  構造的可能性は低い

**Cons (致命的)**:
- ★ **OpenClaw 2026.4.23 の `cron add` に並行起動防止オプションが
  存在しない** (`--help` 確認、§1.4)
- 設計上の安全性を運用設定に依存 = F300 哲学に反する
- 手動起動 + cron 起動の overlap を運用ルールで完全排除するのは
  Fujiwara 兼業前提と矛盾 (認知負荷高、誤操作リスク)
- 本番環境で plist / cron 設定が崩れた場合の検出機構ゼロ
- OpenClaw 仕様が将来「並行起動防止」サポートしたとしても、本案は
  実装当時の OpenClaw 仕様に依存する脆い設計

**修正範囲**:
- `docs/openclaw/agent_registration_guide.md` (運用注意追記)
- 必要なら `scripts/wrapper/price_monitor.py` の docstring に
  「並行起動禁止、cron 1 ジョブのみで起動すること」を明示

**実装工数**: 0.1-0.2 日

## 3. 推奨案選定

### ★ Mac mini 推奨: 案 A (DB schema + PRE-INSERT)

### 根拠 (3 行サマリ)

R1. OpenClaw cron に並行起動防止機能なし (§1.4 確定) → 案 D の前提崩壊。
R2. Stage 3 利確/損切指示の二重送信は致命的 (二重発注の誤解、Fujiwara
    手動発注フローへの直接ダメージ)、構造的解消が必須。
R3. paper_live_notifications PK 制約は既存 → schema 変更最小、
    `INSERT OR IGNORE + rowcount` 判定で原子的予約が SQLite WAL で安定
    動作可能 (実装工数 0.7-1.0 日は許容範囲)。

### 詳細根拠

- **F300 哲学との整合**: F300 Phase 3 で wrapper 群を「ファイル単独で
  safety を担保 = 運用設定に依存しない構造的解消」に再設計した。F300-race
  も同じ哲学を継承すべき → 案 A のみ整合。
- **OpenClaw 仕様の脆弱性**: §1.4 で cron 並行起動防止機能不在を確認、
  仕様変更があっても運用設定に依存する設計は脆い (案 B/D 共通の弱点、
  ただし B は wrapper 内完結で部分的に緩和、D は完全に外部依存)。
- **schema 変更最小**: PRIMARY KEY (position_id, event_type) は既に
  UNIQUE 制約として機能、追加 migration 不要 (`INSERT OR IGNORE` の
  rowcount 判定で原子性確保)。
- **MVP 設計**: status カラム (pending/sent) 拡張は将来拡張余地として
  保持、MVP では「crash = 通知漏れ容認 (Stage 3 LINE 5 部屋疎通の
  別 alert で fallback)」と割り切る選択肢あり。

### 棄却根拠

#### 案 B (filelock) 棄却

- 排他粒度が wrapper 全体 = 並列処理性能ボトルネック (Stage 3 拡張時の制約)
- 「並行起動なし」前提に倒す設計は F300 構造的解消哲学に反する
  (Phase 1 inventory 見落とし §6-20 候補と同型、case を扱わない設計の典型)
- filelock 依存追加は F265 後の依存最小化方針と整合性検証が必要
- stale lock 検出機構 (SIGKILL / kernel panic 時) の追加実装コストが
  案 A の実装工数差分 (0.3-0.5 vs 0.7-1.0 日) を相殺、優位性消失

#### 案 D (運用側責務) 棄却

- ★ **OpenClaw 2026.4.23 cron に並行起動防止オプション不在** (決定打、§1.4)
- 設計上の安全性を運用設定に依存 = F300 哲学に反する
- 手動起動 + cron 起動の overlap を Fujiwara 認知で排除するのは兼業前提
  (誤操作リスク許容)、運用フェイルオーバー時に安全性が崩れる
- plist / cron 設定崩れの検出機構ゼロ → 本番障害時の root cause 特定が
  困難 (silent failure 経路、F271 v1.2 §3.6.12 と同型問題の再来リスク)

## 4. F300 Phase 1 で race condition を見落とした検出経路の事後分析

### 4.1 §6-20 候補: 「inventory 観点の単一レイヤー偏重」

F300 Phase 1 inventory 設計時の観点:

| 観点カテゴリ | 観点項目 | inventory 検出 |
|---|---|---|
| CLI 引数 | --skip-* / sentinel value / required=False | ✅ 検出 (Q1-Q4) |
| Fallback | hardcoded equity / hardcoded entry_price | ✅ 検出 (Q1) |
| Exit code | silent return 0 / sys.exit(1) 不在 | ✅ 検出 (§3.6.12) |
| 構造矛盾 | --symbols 複数受付 + 単一 entry_price | ✅ 検出 (Q2) |
| LIMIT + duplicate | 古い未通知の永久未処理 | ✅ 検出 (Q3) |
| **inter-process race** | SELECT-INSERT 別 tx / lock 設計 | ❌ **見落とし** |
| **inter-wrapper race** | wrapper 間の DB tx 境界 | ❌ **見落とし** |

漏れた観点はすべて「**個別 wrapper 単位の safety hole 列挙では原理的に
検出不可能なクラス**」 = orchestration / inter-process / DB tx 境界。

### 4.2 検出経路 (実際)

1. F300 Phase 3 NEW commit 1 (`5ce8d99`) を Codex pre-commit に投入
2. Codex が CRITICAL 検出: 「price_monitor.py の SELECT と
   MonitoringAlertAgent 側 INSERT が別 tx、並行起動で二重送信できる」
3. HQ-Lean ルール 5 連鎖 (本 F300 内の連鎖 3 件目) → 本部 Q8 判断要請
4. 本部判断 Q8=C: F300-race 別タスク送り (HQ-Lean ルール 5 自己検証
   3 回目結果、scope 別認定)

### 4.3 §6-20 提案 (F271 v1.3 候補)

> Phase 1 inventory 観点に以下を **必須項目** として追加:
> - inter-process race condition (SELECT-INSERT 別 tx, file lock 不在)
> - inter-wrapper race condition (wrapper 間の DB tx 境界, 共有
>   resource 競合)
> - DB transaction boundary race (WAL モード並行書き込み, INSERT OR
>   IGNORE rowcount 判定不在)
> - lock 設計 (排他制御, stale lock 検出)
>
> 検出ツール: Phase 3 commit 前に Codex に「並行起動 race condition」
> 観点を明示要求するレビュープロンプト追加 (`/codex:adversarial-review`
> の必須項目に追加)。

### 4.4 同型問題の既知例

- **F271 §6-14**: multi-symbol + 単一 entry_price 矛盾 (個別観点では
  「両者とも有効引数」だが構造的に矛盾、Codex 検出経由で発覚)
- **本件 (§6-20 候補)**: SELECT-INSERT 別 tx (個別観点では「両者とも
  duplicate 防止に寄与」だが race window で構造的に破綻、Codex 検出
  経由で発覚)

両者とも「**個別観点では正しいが、組み合わせ観点で構造的に破綻**」する
クラスの問題。Phase 1 inventory に **組み合わせ観点 (combination
viewpoint)** を必須項目として組み込む必要あり。

## 5. 本部判断要請事項 (Phase 3 着手前)

### Q1: 案 A の状態カラム拡張範囲

- 案 A-1 (MVP, 推奨): status カラム不要、`INSERT OR IGNORE` の rowcount
  だけで原子的予約。crash 時の orphan は容認 (LINE 5 部屋疎通の別 alert
  で fallback)。
- 案 A-2 (拡張): status (pending/sent) + reserved_at (TTL) カラム追加、
  crash 後 5 分超 pending は別 process が再予約可能。schema migration
  + cleanup ジョブ追加。

### Q2: 既存 `_record_notified` の経路改修方針

- 案 a (推奨): `_record_notified` を `_try_reserve` (INSERT OR IGNORE +
  rowcount return) に rename + 振る舞い変更、`_is_already_notified` を
  廃止 (try_reserve に統合)。
- 案 b: 現関数を維持、新規 `_try_reserve` 追加で並行使用。経路二重化
  (技術的負債候補、現状コードの後方互換テスト維持容易)。

### Q3: price_monitor 側 LEFT JOIN の扱い

- 案 i (推奨): LEFT JOIN 維持 (二重防御、agent 側変更後も SQL レベルで
  事前フィルタ)。
- 案 ii: LEFT JOIN 削除、agent 側 `_try_reserve` のみで重複排除。
