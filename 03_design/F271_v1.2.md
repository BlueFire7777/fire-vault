### §6-9 cache 対象とトリガー経路のデータソース不整合 (v1.2 新規)

cache (similarity_cache / trade_stats_cache 等) と invalidation トリガー
経路 (writer 側 / reset 呼出箇所) のデータソース (テーブル名 / 列名 /
SQL) が不一致だと、cache が古い状態のまま運用継続するリスク。

実証事例:
- F277-A handover §1.2 のテーブル名混同が起点 (positions vs
  paper_live_positions)
- F277残作業 commit 3 で確立、案 Y 採用で trade_stats_cache を
  positions 対象に統一

回避策:
- cache の対象データソース (SELECT 元) と invalidation トリガー経路
  (writer 側 SQL の対象テーブル) を一致させる
- 事前 grep 観点 v3 観点 1-3 で確認 (修正対象関数の読み書きデータソース
  特定 + cache invalidation トリガー経路特定)

### §6-10 handover 記載のテーブル名 / リソース名混同の連鎖 (v1.2 新規)

引き継ぎ文書 (handover) で記載されているテーブル名 / リソース名が実装と
乖離していると、後続作業者が誤った前提で修正を進めて二度踏み抜きリスク。

実証事例:
- F277-A handover §1.2 で「positions テーブル」と記載されていたが
  実装は paper_live_positions テーブル参照、commit 3 Phase 2 で混同が
  連鎖検出

回避策:
- handover 記載のテーブル名 / リソース名は grep で実装と整合確認
- 事前 grep 観点 v3 観点 4 で確認 (流用元の参照対象と流用先の参照対象が
  同レベルの抽象か)

### §6-11 並列処理ライブラリの worker dispatch 仮定の過信 (v1.2 新規)

並列処理ライブラリ (ProcessPoolExecutor 等) の submit 順序が worker
affinity を保証する仮定 (例: submit 1 → worker 1、submit 2 → worker 2)
で実装すると、実態は worker affinity が保証されないため stale cache /
state 問題が発生。

実証事例:
- F277残作業 commit 4 (5cd5e80) で確立、ProcessPoolExecutor の submit
  順序が worker affinity を保証しないことが判明
- マルチプロセス E2E 二重 assert で stale/fresh を構造的に証明

回避策:
- 並列処理ライブラリの worker dispatch は実装挙動で検証 (ドキュメント
  仮定で進めない)
- テスト設計時に Pool 分離 (separate pool per test) を検討

### §6-12 中間報告の行数のみで F タスク紐づけ判断 (v1.2 新規、本部側)

中間報告 / 進捗報告で「+N 行」のような表面情報のみで F タスク紐づけを
判断すると、複数 F 混在の commit を単一 F と誤認するリスク。

実証事例:
- F278 Phase 1 B-3 commit 指示で「+109 行 = F241 単独」と本部側が判断、
  実態は F230 + F241 両方の CLI 拡張混在で B-3a/B-3b 分割となった

回避策:
- 中間報告時点で行数だけでなく diff の論理分割可能性も報告観点に追加
- 事前 grep 観点 v3 観点 5 で確認 (中間報告内容の論理分割可能性、
  複数 F 混在検知)

### §6-13 既存内部機構の API 表面公開時の到達可能性過信 (v1.2 新規、本部側)

既存内部パラメータ (テスト用 skip 引数等) が特定経路 (テスト用直接呼出
のみ) で到達可能だった制約を、新経路 (CLI / API endpoint) から無条件に
公開すると、設計時想定外の経路で副作用が発火する。

実証事例:
- F278 Phase 1 B-3a で LiveAdvisoryChecker の skip パラメータが内部実装
  にあり、CLI 公開承認時に「便利機能」として扱われた → Codex CRITICAL
  検出で構造的解消

回避策:
- CLI / API endpoint の新規引数追加時、引数が背後で操作する内部
  パラメータの「設計時想定到達経路」を確認
- 事前 grep 観点 v3 観点 6 で確認 (API 表面公開時の到達可能性)

### §6-14 Codex CRITICAL 受領時の網羅調査範囲不足 (v1.2 新規、Mac mini側)

Codex CRITICAL 受領後の網羅調査を「同型エラー伝播経路の特定経路のみ」
(例: except 句のみ) で打ち切ると、他経路 (戻り値経由 / 副作用経由) の
同型問題を見逃して二度踏み抜きリスク。

実証事例:
- F278 Phase 1 B-3b で ValueError except の網羅調査時、戻り値経由
  (error_msg) ルートを見逃して Codex 二度目検出

回避策:
- エラー伝播経路を 3 分類 (例外 / 戻り値 / 副作用) で網羅
- 事前 grep 観点 v3 観点 7 で確認 (網羅調査範囲のエラー伝播経路分類展開)

### §6-15 docstring 一次情報による retroactive commit 採用の自明性過信 (v1.2 新規、本部側)

docstring 記載 (「production wrapper」「completed feature」等) を
一次情報として retroactive commit 採用判断すると、内部実装が production
-ready でない (CRITICAL 検出 / safety gate bypass / 偽 PASS 経路) 可能性
を見逃すリスク。

実証事例:
- F278 Phase 1 B-7 で F261 wrapper 群を docstring 記載基準で一括 commit
  承認、Codex CRITICAL 3 件検出で F261-fix 送り

回避策:
- retroactive commit 採用判断は docstring + 内部実装レビュー (事前 grep
  / safety gate 確認) の二段確認
- §3.6.13 適用 (retroactive commit の二段確認)

### §6-16 フェーズ作業性質変化を指示文表現に反映しないテンプレート流用過信 (v1.2 新規、本部側)

フェーズごとの作業性質 (新規 commit 発生 / 既存 commit 監査 / 文書化
のみ等) が異なるが、本部が前フェーズの指示文表現をテンプレート流用すると、
Mac mini 側で実態とのギャップ調整を担うことになる。

実証事例:
- F278 Phase 2 着手時、本部が Phase 1 (新規 commit 発生) の表現
  「retroactive commit + Codex pre-commit 厳格運用」を流用、実態は
  「既存 commit に対する Codex review 経由の遡及監査 + 新規発見
  CRITICAL の修正 commit」だった

回避策:
- フェーズ着手時に作業性質を明文化 (新規 / 監査 / 文書化 / マージ等)
- 指示文表現を実態に合わせる (テンプレート流用避ける)

### §6 章末総括 (v1.2 新規)

### §3.6.12 production CLI のエラー伝播経路と exit code 一貫性 (v1.2 新規)

Stage 3 自動化経路 (OpenClaw / cron / launchd) で呼ばれる production CLI
は、エラー伝播経路 (例外 / 戻り値 / 副作用) すべてに対して exit code
非ゼロを返す責務を持つ。CLI 公開引数は safety gate を bypass しない
(内部 skip パラメータの API 表面公開は §6-13 参照)。

実証事例:
- F278 Phase 1 B-3a (LiveAdvisoryChecker --skip-* CLI 公開削除)
- F278 Phase 1 B-3b (F230 Batch Replay ValueError + error_msg sys.exit(1))
- F261-fix で F261 wrapper 群全般に同型適用予定

実装パターン:
- エラー出力先: sys.stderr 一本化 (stdout との重複削除)
- exit code: 0 = 正常完了、1 = エラー (将来 exit code 細分化検討可)
- 内部 skip パラメータは CLI 公開しない (テスト経路は内部呼出維持)

### §3.6.13 retroactive commit の二段確認 (docstring + 内部実装) (v1.2 新規)

docstring の F タスク紐づけ記載は retroactive commit 候補の必要条件だが
十分条件ではない。内部実装が production-ready かを事前 grep + safety gate
確認で検証する二段確認制を retroactive commit 全般に適用。

実証事例:
- F278 Phase 1 B-7 (F261 wrapper 群、docstring 基準で承認 → CRITICAL 3
  件検出、F261-fix 送り)
- F278 Phase 2 F050 (__init__.py 前方参照、docstring + import 利用実態の
  二段確認で案 b-3 確定)

#### §3.6.13 補強観点: retroactive 遡及監査の記録粒度ルール (v1.2 新規、Phase 4 由来)

retroactive 遡及監査の記録は以下のルールで粒度統一:

##### 修正 commit を発行する場合
- commit message に詳細集約 (Phase / 紐づく F / Codex 検出内容 / 修正
  方針 / 検証結果)
- Vault 記録は不要 (commit message が一次情報、git log で追跡可能)
- 例: F050 案 b-3 commit 74de54e、F051 案 a commit f1c628f、F141 案 b-1
      commit ef97e8e

##### 修正 commit を発行しない場合 (時系列内 fix 済 / 申し送りのみ)
- Vault に handover ファイル作成
  (~/fire-vault/03_design/F___handover_<date>.md)
- §1-§7 の構造化記録 (概要 / 検出内容 / 既存解消 / HEAD 現状 / 監査結論
  / Phase 5 参考価値 / 関連)
- 例: F053_handover_2026-05-05.md

##### 設計判断 / handover (修正に紐づかない記録)
- Vault に handover ファイル作成
- 設計選択肢 + 採択理由 + 関連 commit hash
- 例: F277_B_handover_2026-05-04.md、F142_handover_2026-05-05.md

##### ルール改訂候補 / パターンライブラリ
- Vault に集約ファイル作成 (例: f271_v12_candidates_2026-05-05.md)
- 後続フェーズで正式採用判断対象

### §3.6.14 commit 順序を踏まえた __init__.py import 整合 (v1.2 新規)

retroactive commit (過去コミットの遡及記録) で __init__.py のような統合
点ファイルが後続 commit で追加されるシンボルを前方参照すると、commit
順序的に存在しないシンボルへの依存が発生し、git bisect / 単独 checkout
で動作しなくなる。

実証事例:
- F278 Phase 2 F050 (commit bbdfab2) で __init__.py が F051/F053 の
  シンボルを前方参照、bbdfab2 単独 checkout で import 失敗 → 案 b-3
  (HEAD __init__.py 最小化) で git bisect 実用性回復

検出方法:
- retroactive commit 時に __init__.py の利用実態 (`from package import`
  の grep) を事前確認
- 利用実態ゼロなら __init__.py 最小化 (案 b-3)、利用実態ありなら遅延
  import (PEP 562 __getattr__ 等、案 b-1) で対応

違反時の修正方針:
- HEAD 側で遅延 import 化 / 最小化 → bisect 実用性回復
- 履歴は不可逆扱い、申し送り記録 (案 c) → 現運用実害ゼロなら容認
- git history 改変 (案 a / rebase / force push) → 副作用大、棄却

補強観点 (Phase 2 F050 で確立):
- retroactive commit 時、__init__.py の「将来分まで先取りして」公開
  シンボル設計してしまう罠に注意
- 設計者の意図 (公開 API 設計) と実装の現実 (利用側が submodule 直接
  import を使用) が乖離する場合、__init__.py は前方参照を残したまま
  死蔵コード化する
- 利用実態確認 (`from package import` grep) で死蔵コード判定 → 最小化
  が筋

### 観点 6: CLI / API endpoint の新規引数追加時、引数が背後で操作する

承認したが、Codex CRITICAL で検証ツール偽 PASS リスクを検出。

適用方法:
- CLI / API 新規引数追加時、対応する内部パラメータの設計意図 + 想定到達
  経路を grep + docstring で確認
- 「テスト用」の内部パラメータを CLI 公開しない (内部直接呼出維持)
- safety gate の bypass パラメータは CLI 公開時に追加判断要 (§3.6.12 と
  連動)

### 観点 7: Codex CRITICAL 受領後の網羅調査範囲は「エラー伝播経路の分類」
            (例外 / 戻り値 / 副作用) で展開し、特定経路 (except 句) のみ
            で打ち切らない (v3 新規)

実証: F278 Phase 1 B-3b で ValueError except の網羅調査時、戻り値経由
(error_msg) ルートを見逃して二度踏み抜き。

適用方法:
- Codex CRITICAL 受領後、エラー伝播経路を 3 分類で網羅:
  - 例外経由 (except 句、ValueError 等)
  - 戻り値経由 (error_msg / status / sentinel 値)
  - 副作用経由 (DB rowcount / file existence)
- 各経路で同型問題の有無を確認 → 全経路修正
