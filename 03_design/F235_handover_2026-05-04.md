## 1. F236 申し送り (commit cbb4801)

**該当箇所**: `scripts/emergency_alert.py` の `count_open_positions`

**現状**: `paper_live_positions` の COUNT のみで建玉数を判定。
FIRE 内部 0 件 ≠ 実建玉 0 件 (Stage 3 では楽天証券側に存在する可能性)。

現状は OPERATIONAL_STAGE = m0/m1/m2_plus の場合に
「FIRE 認識 0 件、実建玉は楽天証券で確認してください」
の併記で人間確認を促す暫定対策。

**Stage 3 実装要件**:

- 楽天証券 API から実建玉数を取得
- `paper_live_positions` と合算して total_count を算出
- メッセージから「FIRE 認識 0 件、楽天確認」併記を削除可能 (実建玉直接判定)
- DB エラー時の Stage 別 layer ハンドリング (§3.6.3) は維持

**該当 §**: §3.6.1 Stage 別 layer ハンドリング

## 2. F119 申し送り (commit 8906368)

**該当箇所**: `evaluation/aggregators.py` の各 `_aggregate_*` 関数

**現状**: `paper_live_positions` のみ参照、Stage 3 実建玉は集計対象外。
評価結果 (勝率 / 期待値 / 平均 RR 等) が paper_live のみで算出される。

**Stage 3 実装要件**:

- 楽天証券 API から実建玉履歴を取得
- `paper_live_positions` と実建玉を統合した集計関数を実装
- §3.6.6 後知恵バイアス防止 (`as_of_dt` フィルタ) を実建玉側にも適用
- §3.6.6 TZ 正規化 (UTC 統一) も実建玉側に適用

**該当 §**: §3.6.1 Stage 別 layer ハンドリング (paper_live と実建玉の統合)

## 4. F235 申し送り対象外確認済タスク

F278-Pre 完了タスクのうち、F235 楽天証券連携の影響を受けないと確認済の
タスクを記録。F235 着手時の網羅性確認負荷を軽減する。

### 4.1 F130-F140 Risk 5 件 (commit 7d26777)

**確認結果**: 対象外、F235 楽天証券連携不要。

**根拠**:
- `risk/allowable_loss.py:65` `OperationalStage` Enum で Stage 別係数を実装済
  (paper_live / m0 / m1 / m2_plus でリスク係数変更)
- `get_operational_stage_from_env` で `OPERATIONAL_STAGE` 環境変数経由
- リスク計算は内部係数のみで動作、実建玉履歴への依存なし
- §3.6.1 Stage 別 layer ハンドリングは別文脈 (F236 のような silent suppress
  リスクは該当せず、係数差し替えのみ)

### 4.2 F260 OpenClaw agents docs (commit 614fffd)

**確認結果**: 対象外、F235 楽天証券連携不要。

**根拠**:
- 100% docs ファイル (.md)、実装 (.py) なし
- agent_registration_guide.md + 6 IDENTITY ドキュメント
- Stage 別実装と無関係、実建玉参照なし

### 4.3 F057 Scheduler (commit a4fc339)

**確認結果**: 対象外、F235 楽天証券連携不要。

**根拠**:
- CLI 経由起動の daemon プロセス (PaperLiveScheduler)
- OPERATIONAL_STAGE 判定なし、runner 経由の tick 実行のみ
- 実建玉参照なし、paper_live 専用

**補足**: §3.6.7-9 daemon パターン (SIGTERM ハンドラ / stale tick 防止 /
分割 sleep) を実装済 (CRITICAL 4-8 修正)。launchd 経由起動時の cleanup
保証 + signal 即応性確保。

### 4.4 F058 E2E smoke test (commit f4754c4)

**確認結果**: 対象外、F235 楽天証券連携不要。

**根拠**:
- one-shot E2E smoke テスト (24 時間稼働本番投入前ゲート)
- Stage 別実装と無関係、is_go 判定は paper_live 内部状態のみ参照

**補足**: §3.6.3 silent fallback 廃止 + §3.6.5 is_go 判定基準拡張
(CRITICAL 1-3 修正) を実装済。false GO 出力リスク解消。
