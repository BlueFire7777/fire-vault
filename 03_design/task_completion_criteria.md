## 1. 背景 — 2026-05-04 の 2 つの事故

本仕様書は 2026-05-04 に同日発生した 2 つの事故を起点に制定された。
両事故は表層では別現象だが、根本原因は **「成功の定義が表面的だった」**
点で共通する。

### 事故 1: events=0 事故 (Chain 5h54m 完走の無駄打ち)

- **発生**: 2026-05-03 夜 〜 2026-05-04 朝
- **現象**: `full_chain.sh` (F100 historical → 失敗銘柄補完 → F230
  batch_replay 3run [20/60/120 営業日]) が **5h54m 走行、全 step exit code
  0 で完走**。ところが 3 run すべて `n_events_total=0 / trades=0 /
  positions=0`
- **直接原因**: `features` テーブルが 87 行 / 5 銘柄 / 1 週間しかなく、
  `tick.py:164` の `_has_features` で 4,452 銘柄全件が即 continue → events
  発生不可能 (`market_prices_daily` の 0.0009%)
- **誤報告のリスク**: exit 0 を「成功」と判定すれば、5h54m の無駄打ちが
  本番運用で繰り返される
- **再発防止策**:
  - 技術的: [[F269_chain_assertions|F269]] で Chain assert 4 種 (events>0
    必須 / candidate>0 必須 / entry>0 警告 / coverage>0.5 警告)
  - 制度的: 本仕様書 (F271) で「動いた」と「機能した」を区別

### 事故 2: F233 完了基準甘さ (Wikilink 切れのまま完了判定)

- **発生**: 2026-05-04 朝
- **現象**: F233 (週次レビューテンプレ作成) で、テンプレ内の Wikilink
  (例 `[[F267_features_pipeline|F267]]`) が **リンク先未作成のまま**「完了」
  と Claude が報告。Fujiwara 指摘で 02_todo スタブ 4 件補完作業 (F266 /
  F267 / F268 / F269) を実施し真の完了に
- **直接原因**: Claude が完了基準を矮小化。「ファイル 3 つ作る」を完了条件
  として、「運用可能な状態にする」という本来の完了条件を見落とした
- **誤報告のリスク**: ドキュメント・テンプレ系のタスクで「ファイルが存在する」
  だけで完了とすると、運用開始時にリンク切れが多発し、テンプレが死ぬ
- **再発防止策**: 本仕様書 (F271) で「機能した」の定義に Wikilink 健全性 /
  リンク先存在を含める

### 共通する根本原因

両事故とも、**完了の定義が「動いた」段階で止まっていた**:
- 事故 1: 「Chain が exit 0 で完走した」= 動いた (ただし機能していない)
- 事故 2: 「ファイル 3 つを作成した」= 動いた (ただし運用上は機能しない)

→ 「機能した」「期待値を満たした」を独立して判定する必要がある。

## 2. 完了の 3 段階定義

タスク完了は以下の **3 段階** で判定する。完了報告では到達した段階を
明示すること。

| 段階 | 定義 | 判定方法 | 事故 1 例 | 事故 2 例 |
|---|---|---|---|---|
| **動いた** | 実行が完走した、エラーで止まらなかった | exit code 0 / プロセス正常終了 / ファイルが作成された | Chain が 5h54m 走り終えた | テンプレ md 3 ファイルが作成された |
| **機能した** | 期待される処理が実行され、副作用が発生した | 意味的成功条件 (events>0 等 / Wikilink 健全性 / リンク先存在) | Chain で `events > 0` かつ `candidate > 0` | テンプレ内 Wikilink がリンク切れゼロ、記入例の全リンクが実ファイルを指す |
| **期待値を満たした** | 量・質・基準が事前合意した水準を達成 | KPI / 受入基準 (件数・カバレッジ・性能・規律遵守) | events>0 かつ DD<10% かつ promotion_check 5/5 PASS | Fujiwara が「実運用で使える」と判定、初回運用日に記入できる |

### 重要ポイント

- **3 段階は累積**: 「期待値を満たした」と報告するには、その下位 2 段階も
  満たしていることが前提
- **「動いた」だけで完了報告するのは原則禁止**: 軽微タスク (§ 5-3) を
  除き、最低でも「機能した」段階に到達すること
- **「動いた」と「機能した」の差を必ず言語化する**: 完了報告で「機能した」
  をチェックする際、根拠 (検証コマンド出力 / 件数 / リンク健全性結果)
  を明示する

## 3. タスク種別ごとの完了基準ガイドライン

タスクの性質によって「機能した」「期待値達成」の判定方法が異なる。
以下は主要 4 種別のガイドライン。該当しない種別は最も近いものに準拠し、
完了報告で「該当ガイドライン: 3-X (準拠)」と明記する。

### 3-1. コード実装タスク (例: F267 / F269 / F141 / F211)

| 段階 | 判定 |
|---|---|
| 動いた | ビルド成功 / Python 起動エラーなし / import エラーなし |
| 機能した | 単体テスト or smoke test PASS、想定 I/O が DB に記録される |
| 期待値達成 | 受入基準 (件数 / カバレッジ / 性能 / 既存テスト 0 件回帰) |

**完了報告必須項目**:

- [ ] ビルド/テスト結果 (`pytest` PASS 件数 + 既存への影響 0 件)
- [ ] 単体動作の証跡 (例: `n_events_total=42`、`features 行数 1,074,331`、
  `n_ticks=1340`)
- [ ] 期待値との比較 (例: `events>0 (期待値: >0、達成)`)
- [ ] 既知の制約・残課題 (例: 「announcements 副因 B は本タスクスコープ外」)

**事故 1 の場合の正しい報告例**:

```
## F230 batch_replay 3run 完了報告

### 完了水準: 動いた のみ (機能した に未到達)

- [x] 動いた: Chain が 5h54m 完走、exit 0
- [ ] 機能した: events_total=0 / trades=0 で **未到達**
- [ ] 期待値達成: F053 promotion_check 4/5 FAIL で **未到達**

→ events=0 ブロッカーあり、F267 で features 整備後に再走必要
```

### 3-2. ドキュメント・テンプレタスク (例: F233 / F234 / F270 / F271 自身)

| 段階 | 判定 |
|---|---|
| 動いた | ファイルが作成された (Markdown 構文エラーなし) |
| 機能した | 構文整合性: Markdown lint PASS / Wikilink 健全性 / リンク先存在 |
| 期待値達成 | Fujiwara が「実運用で使える」と判定、運用例で 1 件動作確認 |

**完了報告必須項目**:

- [ ] 作成ファイル一覧 (パス + 行数)
- [ ] **Wikilink 検証結果** (件数 + リンク切れゼロ確認、検証コマンド出力)
- [ ] 記入例 or 動作例 (テンプレなら 1 件記入してみせる)
- [ ] 既存ファイルとの整合性 (frontmatter 規約遵守 / 命名規則準拠)

**Wikilink 健全性の標準検証手順**:

```bash
# 1. 全 Wikilink を抽出
grep -oE '\[\[[^]]+\]\]' <ファイル> | sort -u

# 2. 各リンク先について ls で存在確認
ls /Users/bluefire/fire-vault/<リンク先パス>.md

# 3. プレースホルダ (`<最新ファイル>` 等) を除いて実数カウント
# 期待: 全数機能 (placeholder 以外)
```

**事故 2 の場合の正しい報告例 (F233 真の完了報告)**:

```
## F233 真の完了報告

### 完了水準: 期待値達成

- [x] 動いた: テンプレ 3 ファイル + 02_todo スタブ 4 件 作成
- [x] 機能した: Wikilink 検証 (記入例 5/5、README 6/6 機能化、
      リンク切れゼロ)
- [x] 期待値達成: 初回運用日 (2026-05-10 21:00) に Fujiwara が
      そのまま記入開始可能な状態
```

### 3-3. データ取得・パイプラインタスク (例: F100 / F101 / F268)

| 段階 | 判定 |
|---|---|
| 動いた | スクリプトが完走、exit 0 |
| 機能した | DB に意味のあるレコードが挿入された (件数 > 0、フォーマット正常) |
| 期待値達成 | カバレッジ・鮮度・正確性が要件を満たす (例: Core500 ≥ 95%、
              直近 6 ヶ月期間内) |

**完了報告必須項目**:

- [ ] 取得件数 / 期間 / カバレッジ%
- [ ] 失敗・欠損サンプル (例: 「失敗 6 銘柄: 150A0/15140/.../152A0」)
- [ ] DB 整合性検証 (UPSERT 重複なし、外部キー整合、CHECK 制約 PASS)
- [ ] 取得元の鮮度 (例: 「J-Quants 最新営業日 2026-05-02 取得済」)

**標準検証 SQL** (取得後に必ず実行):

```sql
-- 件数確認
SELECT COUNT(*) FROM <table>;
-- 期間確認
SELECT MIN(date), MAX(date), COUNT(DISTINCT date) FROM <table>;
-- 銘柄カバレッジ
SELECT COUNT(DISTINCT symbol) FROM <table>;
-- 重複チェック
SELECT symbol, date, COUNT(*) FROM <table>
GROUP BY symbol, date HAVING COUNT(*) > 1;
```

### 3-4. ゲート・承認タスク (例: F266 / F241 / F053)

| 段階 | 判定 |
|---|---|
| 動いた | チェックスクリプトが完走、exit 0 |
| 機能した | 全項目に対する判定結果が出た (PASS/FAIL のいずれかが各項目に出る) |
| 期待値達成 | 全項目 PASS or 規定の合格水準を超えた + 「偽 PASS」リスク検証済 |

**完了報告必須項目**:

- [ ] 全項目の PASS/FAIL リスト (例: F053 5 項目、F241 9 項目)
- [ ] FAIL 項目の根拠データ (例: 「sample_size: n_days=0, n_trades=0」)
- [ ] **偽 PASS リスクの検証**: 前提が成立していない PASS を特定する。
      例: events=0 で simulation_accuracy が optimistic_bias_score=0.000
      で「PASS」と出る → 実際にはトレードがないだけの偽 PASS

**偽 PASS の検出ルール**:

- ある項目が PASS でも、その判定根拠となるデータ件数が 0 なら **偽 PASS**
  と扱う
- 例: 「halt_function PASS (halt_reason 0 件)」は、halt 機能が動いた
  証拠がない → 偽 PASS
- 偽 PASS は「期待値達成」段階に **未到達** とする (機能したまでは認める)

## 4. 完了報告テンプレート

タスク完了時は以下フォーマットで報告すること。Markdown のチェックボックス
形式で機械的に判定可能にする。

### 標準フォーマット

```markdown
## F<ID> 完了報告

### セクション 1: 完了水準

- [ ] 動いた: <具体的な達成内容、exit code / ファイル作成等>
- [ ] 機能した: <意味的成功条件、events>0 / Wikilink 健全性等>
- [ ] 期待値を満たした: <KPI / 受入基準達成、Fujiwara 判定等>

### セクション 2: 成果物

- `<ファイルパス>` (行数 X / 件数 X)
- ...

### セクション 3: 検証

#### 3-A. 動作証跡
- ログ抜粋 / コマンド出力 / DB クエリ結果 / スクリーンショット参照

#### 3-B. 完全性チェック
- Wikilink: X/X 機能 (検証コマンド: `grep -oE '\[\[[^]]+\]\]' ...`)
- カバレッジ: X% (検証 SQL: `SELECT ...`)
- テスト: X PASS (回帰: 0 件)

#### 3-C. 既知の制約・残課題
- 制約 1: <記述>
- 残課題 1: <記述、別タスク化推奨なら ID 提案>

### セクション 4: 次タスクへの引き継ぎ

- ブロック解除タスク: F<ID> (本タスク完了で着手可能)
- 派生起票: F<ID> <タイトル> (新規タスクとして TODO_Master 追加候補)
- 関連 Wikilink: [[<ノート>]]
```

### 軽量フォーマット (軽微タスク向け、§ 5-3 参照)

```markdown
## F<ID> 完了 (軽微タスク自己申告)

- [x] 動いた + 機能した: <一文サマリ + 検証コマンド出力>
- 成果物: `<ファイルパス>` (差分 X 行)
- 制約: なし / <記述>
```

## 5. レビュー・二段確認制

タスクの重要度に応じてレビュー段数を変える。

### 5-1. 重要タスク (Stage 移行ブロッカー / 最優先 / 不可逆操作)

該当例: F266 (Stage 3 ゲート) / F241 / F053 / DB 破壊系 / 本番デプロイ

**フロー**:

1. Mac mini Claude Code 側 完了報告 (§ 4 標準フォーマット)
2. Claude.ai 側で完了基準レビュー (3 段階のどこに到達したか確認、偽 PASS
   検証)
3. Fujiwara 最終判定 (`--fujiwara-accept` 等の明示承認)

### 5-2. 通常タスク (優先度 高 / 中、工数 0.5 〜 数日)

該当例: F267 / F230 / F233 / F271 自身

**フロー**:

1. Claude 側 完了報告 (§ 4 標準フォーマット)
2. Fujiwara が任意でレビュー (5 分程度の目視確認)

### 5-3. 軽微タスク (工数 0.5 日未満 / リスクなし / 副作用なし)

該当例: 誤字修正 / コメント追加 / .gitignore 追記 / log.md の type 別集計

**フロー**:

1. 自己申告のみで完了 (§ 4 軽量フォーマット可)

**判定基準**:
- 工数 < 4 時間
- 既存ファイル削除 / DB 変更 / 外部送信 を含まない
- 失敗しても 5 分で revert 可能

該当しない場合は § 5-2 通常タスクに昇格。

## 6. アンチパターン集 (やってはいけない完了報告)

### 6-1. 「ファイルを作りました」だけ報告

**NG 例**:

> F233 完了。`weekly_review_template.md` (250 行) 作成しました。

**原因**: 「動いた」のみ報告、「機能した」を確認していない。Wikilink
切れがあっても気付かない。

**正例**:

> F233 完了。テンプレファイル作成 + 記入例 + Wikilink 検証
> (5/5 機能化、リンク切れ 1 件 [`06_monthly/2026-05` 系 Wikilink] →
> テキスト表記に修正済み)。

### 6-2. 「exit code 0」を成功と見なす

**NG 例**:

> Chain 完走、exit 0 で成功です。

**原因**: 「動いた」と「機能した」を混同。意味的成功条件 (events / trades /
positions の件数) を確認していない。

**正例**:

> Chain 完走 (exit 0) しましたが、`events_total=0` のため「機能した」
> 水準には未到達。仮説 A 〜 D の切り分けが必要です。
> → 仮説 A 確定 (features 87 行のみ、F267 で対応)

### 6-3. 「将来作るので OK」で先送り

**NG 例**:

> リンク先は将来作成すれば良いので完了とします。

**原因**: 完了基準を矮小化、運用開始時点で動かないものを納品。「動いた」
段階で止めて「機能した」を放棄している。

**正例**:

> リンク先 4 件 (F266 / F267 / F268 / F269) が未作成のため、本タスクは
> 未完了。スタブ作成を追加実施します。
> → スタブ 4 件作成完了、Wikilink 検証 5/5 機能化、真の完了。

### 6-4. 「動作確認不要なほど自明」

**NG 例**:

> ファイル名を変えただけなのでテスト不要、完了です。

**原因**: 自明性の過信、関連 import やリンクの破壊リスクを軽視。

**正例**:

> ファイル名変更後、`grep -r "<旧ファイル名>" ~/fire-vault/` で参照箇所
> 4 件確認、すべて更新済み。Wikilink も `[[新ファイル名]]` に置換、
> リンク切れ 0 件。動作確認 PASS。

### 6-5. 「テスト PASS」だけで「機能した」と判定

**NG 例**:

> 単体テスト 20/20 PASS、F267 完了です。

**原因**: テストはモック前提で PASS することがある。実 I/O / 実 DB での
動作確認 (smoke test) が必要。

**正例**:

> 単体テスト 20/20 PASS。real DB smoke で `extract_features --scope
> core500 --from 2025-11-01 --to 2026-05-02 --parallel 12` 実行、
> features 行数 87 → 1,074,331 (+1,074,244)、err=0、Run a に必要な
> coverage 達成を確認。

### 6-6. 「全項目 PASS」だけで偽 PASS を見逃す

**NG 例**:

> F053 promotion_check で 1/5 PASS、`simulation_accuracy=PASS` です。

**原因**: optimistic_bias_score=0.000 は events=0 由来の偽 PASS。前提が
成立していない PASS を見逃している。

**正例**:

> F053 promotion_check 1/5 PASS だが、PASS した `simulation_accuracy`
> は events=0 由来の **偽 PASS** (optimistic_bias_score=0.000、評価対象
> なし)。実質 0/5 PASS と扱う。

### 6-7. 「線形外挿で楽観視」 (v1.1 追記、F273/F274 事故から制定)

**NG 例**:

> 13 件の patterns で per-call 8 ms (smoke test 完走)。
> 4,700 件にしても線形に増えるはずだから、Run a 全体 91 分 × 360 倍 =
> 大丈夫でしょう、Phase 2-B 進行します。

**原因**: 小規模試走の所要時間を線形外挿し、性能オーダーの破綻を検証
していない。実際は O(N) スキャンで 4,700/13 = 360 倍 = 21 日に膨張する
ケースを想定外にしている。

**事例**:

- **F273 Phase 2-C** (2026-05-04 朝): patterns 13 件で per-call 0.58 ms
  を確認 → 4,700 件にしたら 2,733 ms/call、Run a 全体 21 日見積と判明、
  走行前中止
- **F274 Phase 2-D** (2026-05-04 夕): SimilarityEngine 単独 1.5 ms から
  「Run a 17 分」と外挿 → 実 ReproducibilityEngine 経由は 9.81 ms、Run a
  102 分で中止

**正例**:

> 13 件 per-call 0.58 ms。4,700 件で **理論的に 2,733 ms/call** (O(N) 線形
> スキャンの場合) → Run a 全体 21 日見積。**走行前中止**、最適化必須。
> 段階試走 (10 → 100 → 500 件) で per-call が銘柄数依存 (線形 N) か独立
> (定数) かを実測してから本走行する。

**原則**:

- 性能オーダー (O 記法) を **理論計算してから** 実装に入る
- 段階試走で per-call の **線形性** (10 銘柄時 ≒ 100 銘柄時 ≒ 500 銘柄時)
  を確認、±20% 超過は警告
- 実用可能ライン (Run a 90 分以内 等) を事前定義し、未達なら別案検討

### 6-8. 「計測対象の取り違え」 (v1.1 追記、F274 事故から制定)

**NG 例**:

> SimilarityEngine.search() の per-call 1.5 ms 達成、Run a 17 分で完走
> 確実です。

**原因**: 計測対象が運用で支配的な経路ではなく、サブモジュール単独。
本番ループで呼ばれるのは ReproducibilityEngine.evaluate() であり、
SimilarityEngine.search() はその一部に過ぎない。

**事例**:

- **F274 Phase 2-A/B/C** (2026-05-04 夕): SimilarityEngine.search() の
  per-call 1.5 ms を測って合格判定 → 実 Run a で per-tick 4.57 秒 (期待
  2.5 秒の 1.83 倍) で中止判断。本来計測すべきは ReproducibilityEngine.
  evaluate() 経由 (9.81 ms)、追加 SQL (fetch_pattern_trade_stats × 5 +
  sector_filter × 4,700) を考慮していなかった

**正例**:

> ReproducibilityEngine.evaluate() (運用経路) で cProfile 内訳測定:
> per-call 9.81 ms、内訳 SQL 40 回 + sector_filter Python loop 4,700 回。
> SimilarityEngine 単独 (3.14 ms) は補助情報として扱い、F275 で
> 運用経路のボトルネックを特定して -7.58 ms 削減 → 2.23 ms/call 達成。

**原則**:

- **最終的に運用で呼ばれる経路** で計測する (本タスク以外で呼ばれる
  サブモジュール単独計測は補助情報のみ)
- エンドツーエンドの実測値で目標達成判定する
- 計測前に「**この計測値は本番で実際に支配的な経路の値か**」を自問する
- cProfile などのプロファイラで内訳を取り、隠れたコストを発見する
