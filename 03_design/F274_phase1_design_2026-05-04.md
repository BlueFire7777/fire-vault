### 2-2. patterns / features の現状

| テーブル | 件数 | 既存インデックス |
|---|---|---|
| patterns | **4,709** (= 旧 9 + F273_BREAKOUT_* 4,700) | sqlite_autoindex (pattern_id) / idx_pat_layer_rank / idx_pat_status / idx_pat_parent / idx_pat_lane / idx_pat_strategy / idx_patterns_sector_filter / idx_patterns_symbol_whitelist |
| features | 1,074,331 | sqlite_autoindex / **idx_feat_symbol_dt** (symbol, dt) |

→ features.idx_feat_symbol_dt は既存で機能している。問題は **per-pattern で 2 SQL 呼ぶこと**。

### 2-3. per-pattern 0.58 ms の内訳 (実測)

**100 patterns sample で測定**:

| 項目 | 100 件あたり | 4,709 件推定 | 比率 |
|---|---|---|---|
| **SQL fetch_pattern_feature_vector (2 SQL)** | 60 ms | **2,833 ms** | **98.4%** |
| Python similarity_score | 0.5 ms | 22 ms | 0.8% |
| SELECT * FROM patterns | (1 回のみ) | 9 ms | 0.3% |
| **合計** | — | **2,864 ms** | (実測 2,733 ms と誤差 5%) |

**結論: ボトルネックは SQL の繰り返し呼び出し (per-pattern 2 SQL × 4,709 = 9,418 query/search())**。Python 計算は 1% 以下。

## 3. STEP 1-B: 最適化 3 案比較 (実測ベース)

### 3-1. 比較表

| 案 | 概要 | 期待 per-call | 1 tick × 500 銘柄 | 1340 tick 全体 | 工数 | リスク |
|---|---|---|---|---|---|---|
| 現状 | O(N) 全件スキャン × 2 SQL/pattern | 2,733 ms | 22.8 分 | **21.2 日** | — | (不可) |
| **P3-A 強化版 (推奨)** | キャッシュ + numpy 行列演算 | **約 1 ms** | **0.5 秒** | **約 11 分** | 6h | numpy 依存追加 / cache 無効化 |
| P3-A pure (副案) | キャッシュ (pure Python) + 1 SQL load | 26.6 ms (実測) | 13.3 秒 | **5 時間** | 4h | 目標未達、夜間投入用途のみ |
| P3-B (劣) | SQL JOIN 化のみ (キャッシュなし) | 推定 70-100 ms | 35-50 秒 | **14 時間** | 2h | 目標未達、不採用 |
| P3-C (劣) | numpy のみ (キャッシュなし) | 推定 50-80 ms | 25-40 秒 | **9-12 時間** | 3h | 起動毎に numpy load = 効果薄、不採用 |

### 3-2. 各案の詳細根拠

#### **案 P3-A 強化版 (推奨)**: キャッシュ + numpy 行列演算

実装方針:
1. **CachedSimilarityEngine** クラス新規 (シングルトン)
2. 起動時 1 回のみ:
   - 1 SQL JOIN: `SELECT p.pattern_id, p.layer, p.status, f.feature_key, f.value FROM patterns p INNER JOIN features f ON ...` (実測 51 ms / 84,600 行)
   - Python 側で group by → numpy 配列 (shape = (4,709, 18))
3. per-call:
   - vec_now を numpy 配列化: 1 ms 未満
   - 一括距離計算: `np.abs(matrix - vec_now).clip(0,1).mean(axis=1)` (75,200 算術 = 1 ms)
   - sort + top_n: `np.argpartition` で 1 ms 未満
   - **合計約 1 ms/call**

メモリ: 4,709 × 18 × 8 byte = 約 680 KB (許容範囲内)

#### 案 P3-A pure (副案、numpy なし)

pure Python キャッシュで実測:
- 起動時 load: 67 ms (1 SQL + group)
- per-call: **26.6 ms** (4,709 件の `similarity_score` Python loop)
- 1 tick × 500 銘柄: 13.3 秒
- 1340 tick: **約 5 時間**

→ 目標 90 分は超えるが、夜間投入で Run a 単独走行可能。numpy が後で導入できない場合の安全網。

#### 案 P3-B (SQL JOIN 化のみ、不採用)

実装: SimilarityEngine.search() 内で 1 大 JOIN クエリに集約

実測 JOIN: 51 ms。per-call では vec_now 取得 + JOIN + Python sim = 約 70-100 ms。

→ 1 tick × 500 = 35-50 秒、1340 tick で 14 時間。**90 分目標未達、不採用**。

#### 案 P3-C (numpy のみ、不採用)

毎 search() 呼び出しで numpy 配列を構築 → 起動時の load コスト (50-100 ms) が毎回かかる。キャッシュ無いと **逆に遅くなる**。

→ **不採用** (キャッシュ前提なら P3-A 強化版に集約)。

## 4. STEP 1-C: 採用案の性能オーダー事前計算 (理論 + 実測)

### 4-1. 採用: 案 P3-A 強化版 (numpy + キャッシュ)

### 4-2. 期待 per-call 時間 (理論計算)

| 段階 | 時間 |
|---|---|
| vec_now SQL (`SELECT feature_key, value FROM features WHERE symbol=? AND dt=?`、idx_feat_symbol_dt 利用) | 1 ms (実測) |
| vec_now → numpy 配列化 | < 0.1 ms |
| `np.abs(matrix - vec_now)` broadcasting (4,709 × 18 = 84,762 算術) | < 0.5 ms |
| `.clip(0,1).mean(axis=1)` | < 0.3 ms |
| Layer/Status boost vectorize | < 0.1 ms |
| `np.argpartition(-scores, top_n)` | < 0.5 ms |
| 結果 dict 化 (top_n=10 のみ) | < 0.1 ms |
| **合計** | **約 2-3 ms/call** |

実測ベースのバッファ込みで **5 ms/call** 想定。

### 4-3. Run a 全体走行時間 (案 P3-A 強化版)

| 段階 | 値 |
|---|---|
| per-call | 5 ms (バッファ込み) |
| 1 tick × 500 銘柄 | 2.5 秒 |
| 1340 tick | **約 56 分** ✅ 目標 90 分以内 |

### 4-4. 副案 P3-A pure (numpy なし) の性能

実測 per-call 26.6 ms → 1340 tick で 5 時間 → **目標未達**。Fujiwara が numpy 依存を許容しない場合の代替。

### 4-5. 線形外挿楽観視の回避 (F271 § 6-5)

Phase 2-A → 2-C で発覚した「13 件で 91 分 → 4,700 件に外挿せず楽観視」アンチパターンを回避するため、**Phase 2 着手後も段階試走 (10 → 100 → 500 銘柄) で per-call 線形性を確認**する設計。

具体的合格基準 (Phase 2-A/B/C):
- Phase 2-A (10 銘柄): per-call 5 ms 以内 / 全体 50 ms 以内
- Phase 2-B (100 銘柄): per-call 5 ms 以内 / 全体 500 ms 以内 (10 倍線形)
- Phase 2-C (500 銘柄): per-call 5 ms 以内 / 1 tick 2.5 秒以内 (50 倍線形)

未達時は **Phase 2-D (Run a) には進まず Fujiwara 報告**。

## 8. 主要発見・判断 (5 点)

1. **per-pattern 0.58 ms の 98.4% は SQL fetch_pattern_feature_vector**: 1 pattern あたり 2 SQL × 4,709 件 = 9,418 query/search() が真のボトルネック
2. **案 P3-A 強化版が唯一目標達成可能**: numpy + キャッシュで per-call 2-3 ms、Run a 約 56 分 (90 分以内)
3. **pure Python キャッシュ (副案 P3-D) は 5 時間で実用境界**: numpy 不可時の fallback として保留可能、ただし Run b/c 連続走行は厳しい
4. **線形外挿楽観視の再発防止**: Phase 2-A/B/C で段階試走 + 各段階合格基準 (per-call 5 ms 以内 / 線形性 ±20%) を事前定義、未達なら Phase 2-D (Run a) には進まない
5. **Fujiwara 二段確認 (F271 § 5)**: Phase 1 完了時点でレビューを挟む (本レポート)、numpy 依存と段階試走合格基準への同意を取得後に Phase 2 着手
