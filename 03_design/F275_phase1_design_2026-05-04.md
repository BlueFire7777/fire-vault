## 3. STEP 1-B: target_patterns 実体調査結果

### 3-1. target_patterns の実体 (確定)

- `tick.py:51` で `TickContext.target_patterns: list[str]` 定義
- `simulation/paper_live/batch_replay.py:220` で `start_session(target_patterns=patterns)` 引数として記録
- **`extract_candidates()` は target_patterns を参照しない**
- **`ReproducibilityEngine.evaluate()` / `SimilarityEngine.search()` は target_patterns 引数を受けない**

### 3-2. 結論: target_patterns はメタデータ専用

- Run a 中の SimilarityEngine は **patterns 全件 (4,709、F273_BREAKOUT_* 4,700 件含む) を対象**としている
- target_patterns に "1 件のみ" と記録されていても、実評価には影響しない
- → **F273 events=0、F274 Phase 2-D 中止の真因は target_patterns ではない**
- 「target_patterns 1 件のみ」は F273 / F274 報告時の誤検知 (実害なし)

### 3-3. F275 スコープ 5 (target_patterns 仕様確認・修正): **対応不要**

evaluation 経路に影響しないため修正不要。記録の意味では `start_session(target_patterns=None)` で active 系 (= approved_active + watchlist) を入れる現仕様で合理的。F275 スコープ 5 は **対応不要 (調査済) として完了**。

### 4-3. **重大な制限事項発見**: events_total >= 50 達成は F275 スコープ外

#### 現状の score 構成 (実測 0.5787 の内訳)

72030/2026-04-30 で `decision="reference_only"` / `score=0.5787`:

```json
{
  "components": {
    "similarity": 1.0,         // ← max 達成
    "win_rate": 0.5,           // ← positions 空 default
    "expected_value": 0.5,     // ← positions 空 default
    "regime_fit": 0.5,         // ← regime collector 未稼働 default
    "fill_quality": 0.5,       // ← default
    "priority_boost": 0.1867,  // ← active=1 件のみ
    "risk_fit": 0.7
  },
  "weights": {
    "similarity": 0.2,
    "win_rate": 0.25,
    "expected_value": 0.2,
    "regime_fit": 0.1,
    "fill_quality": 0.1,
    "priority_boost": 0.1,
    "risk_fit": 0.05
  }
}
// score = 0.2*1.0 + 0.25*0.5 + 0.2*0.5 + 0.1*0.5 + 0.1*0.5 + 0.1*0.1867 + 0.05*0.7
//       = 0.20 + 0.125 + 0.10 + 0.05 + 0.05 + 0.01867 + 0.035 = 0.5787 ✓
```

#### decision="execute" 到達条件

```
n_similar >= 5 AND score >= 0.65
```

現状 sim=10、score=0.5787 → score を **+0.07** 上げる必要。

#### 上昇可能な components (positions / regime に依存)

| component | 現状 | 上限 | 上昇分 score 寄与 |
|---|---|---|---|
| win_rate | 0.5 (positions 空) | 1.0 | +0.5 × 0.25 = +0.125 ← **これだけで execute 達成可** |
| expected_value | 0.5 (positions 空) | 1.0 | +0.5 × 0.2 = +0.10 ← これでも execute 達成可 |
| regime_fit | 0.5 (regime 未稼働) | 1.0 | +0.5 × 0.1 = +0.05 ← 単独では未達 (0.6287) |
| fill_quality | 0.5 (default) | 1.0 | +0.5 × 0.1 = +0.05 |
| priority_boost | 0.1867 (active=1) | 推定 1.0 | +0.81 × 0.1 = +0.081 ← active 拡大で達成可 |

→ **win_rate / expected_value / priority_boost のいずれかを上げれば execute 達成可能**

#### F275 スコープでこれらは上げられるか?

| 改善経路 | F275 スコープ | 別タスク |
|---|---|---|
| positions テーブル seeding (win_rate / expected_value 稼働) | ❌ スコープ外 | F276 候補: Backtest 経由 positions seeding |
| F104 (regime collector 解禁) | ❌ スコープ外 | F104 既起票 |
| Layer 3 active 拡大 (priority_boost 強化) | ❌ スコープ外 | F277 候補: 規律 R-13-08 厳守の上で承認制 |

→ **F275 だけでは events_total >= 50 達成は構造的に不可能**

### 4-4. F275 のスコープ修正提案

**修正前 (元スコープ)**:
- 性能目標 + events_total >= 50 達成

**修正後 (推奨)**:
- 性能目標のみ (per-call 5 ms 以内、Run a 60 分以内)
- events_total >= 50 は **F276 (新規)** として別タスク起票
  - スコープ: positions seeding (Backtest 経由 or 手動) + F104 (regime) + Layer 3 拡大
  - 工数: 2〜3 日 (Backtest seeding は F040 既実装活用)

## 7. 主要発見・判断 (5 点)

1. **per-call 9.81 ms の内訳特定**: SQL 40 回/call が主要因 (signature 5 + fetch × 10 + trade_stats 5 + その他)、sector_filter Python loop 4,700 回/call が副要因
2. **target_patterns はメタデータ扱い**: F273/F274 報告時の「1 件のみ問題」は誤検知、評価本体に影響なし、F275 スコープ 5 対応不要
3. **score 0.5787 頭打ち理由確定**: positions 空 (win_rate=0.5/expected_value=0.5 default) + regime 未稼働 (regime_fit=0.5 default) + active=1 件 (priority_boost 弱) のため、SCORE_THRESHOLD_EXECUTE=0.65 に届かない構造
4. **events_total >= 50 達成は F275 スコープ外**: 性能改善しても decision="execute" 到達不能。F276 (新規) で positions seeding + F104 + Layer 3 拡大が必要
5. **F275 スコープ修正提案**: 性能目標のみに絞り、events 達成は F276 移管。工数 6〜8 時間に短縮、明確な完了条件で F271 § 6-3「将来作るので OK」アンチパターン回避
