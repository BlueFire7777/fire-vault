## 7. signature 機構 + 案 Y 採用記録 (commit 3 (C))

F277-B commit 3 (C、本タスク <commit ハッシュは追記時点未確定>) で
trade_stats_cache に signature 検証機構を追加。

### 採用案 Y (signature = positions テーブル対象)

- **F275 設計意図 (cache=positions、reset=positions writer) と整合**
- signature と cache 対象が同テーブル → 論理整合性最高
- similarity.py `_patterns_signature` (5 SQL 分割) と異なり、
  positions は 1 テーブル + status 2 値のみのため **1 SQL atomic
  snapshot** に集約 (判断 a 採用、Mac mini 判断)

### signature 構成 (5 要素 tuple)

- `COUNT(*)`: 総件数 (INSERT 検知)
- `MAX(exit_dt)`: close 状態変化検知 (positions に updated_at なし)
- `MAX(created_at)`: 新規 INSERT 時刻検知
- `status='open'` COUNT
- `status='closed'` COUNT

`SUM(pnl)` 含めない (判断 a): similarity.py 同型 (COUNT + MAX 系のみ、
数値 SUM なし)、pnl 値のみ更新は backtest 経路で実用上稀。

### TTL 運用切替

- `DEFAULT_TRADE_STATS_TTL_SEC = 0.0` (機構導入完了、後方互換 100%)
- TTL > 0 を明示指定したときのみ signature 検証が有効化
- 運用切替判断は commit 4 統合フェーズ or Stage 3 移行時に Fujiwara
  承認制で確定
- 実測値取得方法: Stage 3 paper_live で stale cache 発生頻度測定 →
  TTL 値決定

### similarity.py との非対称性 (commit 4 申し送り)

- similarity.py: 5 SQL 分割 (patterns 多次元集計のため)
- reproducibility.py: 1 SQL 集約 (positions 1 テーブル + status 2 値)
- commit 4 統合で similarity.py 側 atomic snapshot 化検討候補
  (CASE WHEN 多用での集約可否、メンテナンス性とのトレードオフ評価)
