## §6-9 cache 対象とトリガー経路のデータソース不整合

### 問題

「writer 側に invalidation を統合した = cache 整合性が取れた」と早合点する
認知バイアス。cache 対象データソース (cache 関数が読み取るテーブル / API)
と invalidation トリガー経路の書き込みデータソースが**異なるテーブル**を
扱っているにもかかわらず、関数名や handover 記載のテーブル名混同で同一と
扱ってしまう。

### 発端 (具体事例)

- F277-A handover §1.2 line 64「cache 対象: positions の trade_stats」
  記載は F275 commit message 通りで正しい (cache=positions)
- ただし PositionTracker は **paper_live_positions** writer であり、
  cache 対象テーブル (positions) と異なる
- F277-B commit 2 で PositionTracker._insert/_update に
  `reset_trade_stats_cache()` 統合 → cache 対象 (positions) と
  reset トリガー (paper_live_positions writer) のデータソース不一致

### 連鎖

- F277-B commit 2 Phase 1: Mac mini が PositionTracker = trade_stats
  writer と暗黙的に扱う、Phase 1 中間報告本部承認
- F277-B commit 3 Phase 2: signature SQL を書く段で発見、Phase 2 中間
  報告で本部に判断要請

### 巻き戻し

- 本部指示 Step A/B/C で F275 commit message + handover 記載の矛盾を
  事実抽出
- 案 Y (signature = positions) 採用、F275 設計意図に整合
- commit 2 の reset 統合は将来形先行として暫定維持 (案 c、§9 申し送り)

### 防止策 (Phase 1 事前 grep 標準観点 v2)

1. 修正対象関数の読み取りデータソース (テーブル名 / ファイルパス) 特定
2. 修正対象関数の書き込みデータソース特定
3. cache / state / 副作用が絡む場合: cache 対象データソースと
   invalidation トリガー経路のデータソースが一致するか確認
4. 既存実装からの流用提案時: 流用元の参照対象と流用先の参照対象が
   同レベルの抽象か確認

### F271 v1.2 への組み込み候補位置

第 6 章「不明 / 曖昧時の対応」または新規第 7 章「Phase 1 事前 grep 標準
観点」として配置検討。

## §6-10 handover 記載のテーブル名 / リソース名混同の連鎖

### 問題

handover (引き継ぎ) ファイルに記載されたテーブル名 / 関数名 / リソース名
が、実コードと正確に対応しているか確認せずに記載すると、後続タスクの実装
で同じ混同が再生産される (連鎖)。引き継ぎ受け手は handover 記載を信頼
する傾向があるため、混同は気付かれずに伝播する。

### 発端 (具体事例)

F277-A handover §1.2 のテーブル名混同記載:
- 「cache 対象: positions の trade_stats」(正しい記載) を
- PositionTracker (= paper_live_positions writer) 文脈で記載した結果、
- 後続の commit 2 で「PositionTracker = positions writer」と誤読される
  起点になった

### 影響範囲

- F277-B commit 2 (7d5d049): 誤読を反映した実装、本部承認を得て push
- F277-B commit 3 Phase 1: 誤読が継続、案 X (signature = paper_live_positions)
  を Phase 1 中間報告で推奨
- = 2 コミット分の混同連鎖

### 防止策 (本日確立)

handover 作成時:
- テーブル名 / 関数名は grep で実コードと照合してから記載
- 「cache 対象テーブル」と「writer 経路のテーブル」を分けて明記、
  混同が起きないよう構造化

引き継ぎ受け手:
- handover 記載のテーブル名 / 関数名を信頼せず、Phase 1 で grep 再確認
- 特に cache / state / 副作用が絡む場合は §6-9 観点 v2 を適用

### F271 v1.2 への組み込み候補位置

第 5 章「タスク引き継ぎ規約」または新規 handover 作成ガイドラインとして
配置検討。
