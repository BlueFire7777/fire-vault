## 実装メモ

### PatternResearchAgent.register_proposal の API 確認結果

正規ルートとして `agent.register_proposal(proposal, approved_by="...")` を採用。
内部で `PatternStore.register_candidate` を呼び、Status=CANDIDATE / Layer=ARCHIVE (1)
に置く。**Fujiwara 確定方針の Layer=2 (VALID_LIBRARY) には届かないため、
登録後に SQL UPDATE で bump** する 2 段階方式とした:

1. `propose_new_candidate` → `CandidateProposal` (セグメント検証 + pattern_id 生成)
2. `register_proposal(proposal, approved_by="seed_script_2026-05-02")` (audit trail)
3. `UPDATE patterns SET layer=2, change_reason='Initial seed...'` (bump)

冪等性: `register_proposal` が `status="already_exists"` を返すので 2 回目以降の実行は
全件スキップ。

### F054 extract_candidates 動作確認

```python
extract_candidates(ctx) → 0 candidates  (期待通り)
```

理由: 8 つの seed patterns は `symbol_seed="0000"` で登録 → features テーブルに
"0000" の特徴量はない → ReproducibilityEngine.search で類似マッチが見つからない
→ 候補抽出 0 件。

これは想定動作:
- patterns × features の両方が揃って初めて CANDIDATE が出る設計 (F054)
- 実銘柄に紐付くパターンが必要 → 次ステップで実銘柄向けに features + patterns
  を蓄積する運用フェーズに移る
