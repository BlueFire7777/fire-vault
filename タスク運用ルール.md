## 7. Vault Git 運用ルール (2026-05-03 確立)

### 基本方針

`~/fire-vault` は **GitHub: `BlueFire7777/fire-vault` (private, main branch)** で管理する。
Mac mini 障害時の復旧と Karpathy 式 `log.md` の時系列追跡のため、Vault 更新は必ず GitHub に push する。

### 必須ワークフロー

Vault に変更を加える全タスクで以下を完了条件とする:

1. ファイル編集 (要件書追記 / 新章作成 / log.md 追記 / TODO メモ等)
2. `git add -A` (または変更ファイルを明示)
3. `git commit -m "<type>: <scope> ..."` (Conventional Commits 形式)
4. **`git push origin main`** ← これを忘れない
5. `git log -1` で最新コミット確認

### コミットメッセージ規約

- `docs:` 要件書・design ノート・log.md 等のドキュメント変更
- `feat:` 新機能 (主に code 側で使用、Vault では稀)
- `fix:` 誤記訂正・矛盾修正
- `chore:` `.gitignore` / 設定ファイル等のメタ変更

例:
- `docs: v3.4 空売り対応パッチ適用 (12章追記 + 第42章新設)`
- `docs(02_todo): F230 完了メモ追加`
- `docs(04_daily): 2026-05-03 の作業ログ追加`
- `chore: .gitignore に secrets パターン追加`

### 分割コミット原則

- 設定変更とコンテンツ変更は別コミット
- 複数 F-task のメモは backfill として別コミット
- v3.x 等の仕様変更は単独コミット
- 日次の daily ノートも別コミット

### .gitignore で除外しているもの

- `.env` / `.env.*`
- `*.key` / `*.pem`
- `*credential*` / `*token*` / `*.secret`
- `.obsidian/workspace*` (個人の表示状態)
- `.DS_Store`
- `.claude/` (kepano/obsidian-skills のクローン、別リポジトリ管理)

### 機密情報チェック

新規ファイル追加時、コミット前に以下で機密混入を確認:

```bash
git ls-files | grep -iE '\.env|credential|token|secret|password|jquants_api'
```

ヒットした場合は `git rm --cached` + `.gitignore` 追記で対処、push しない。

### 復旧手順 (Mac mini 障害時)

1. 新環境で `git clone https://github.com/BlueFire7777/fire-vault.git ~/fire-vault`
2. Obsidian で `~/fire-vault` を Vault として開き直す
3. `.gitignore` で除外されている `.env` 系は別バックアップから復旧 (1Password 等)

## 8. タスク完了基準 (2026-05-04 v1.0 / F271 制定)

タスク完了の判定は **3 段階** で行う:

1. **動いた** — 実行が完走、エラーで止まらなかった (exit code 0 / プロセス
   正常終了 / ファイル作成成功)
2. **機能した** — 期待される処理が実行され、副作用が発生 (events>0 / Wikilink
   健全性 / リンク先存在 / 単体動作の証跡)
3. **期待値を満たした** — 量・質・基準が事前合意した水準を達成 (KPI / 受入
   基準 / Fujiwara 「実運用で使える」判定)

完了報告時は **到達した段階を明示** すること。詳細は
[[03_design/task_completion_criteria|タスク完了基準仕様書]] を参照。

### 重要原則 (短文版)

- **exit code 0 = 成功 ではない** (events / trades / 件数等の意味的成功
  条件を必ず確認)
- **「将来作るので OK」は完了ではない** (リンク先未作成のテンプレは未完了)
- **「動いた」までを「機能した」と誤報告しない**
- **テスト PASS だけで「機能した」と判定しない** (実 I/O / 実 DB の smoke
  も必要)
- **「全項目 PASS」でも偽 PASS (前提不成立) を別途検証する**

### 完了報告フォーマット (4 項目必須)

詳細は [[03_design/task_completion_criteria|タスク完了基準仕様書]] § 4
「完了報告テンプレート」を参照。最低限以下 4 項目を必須とする:

1. **完了水準** (3 段階のチェックボックス、各段階の達成内容明示)
2. **成果物** (ファイルパス + 行数 / 件数)
3. **検証** (動作証跡 + 完全性チェック [Wikilink / カバレッジ等] + 既知の
   制約・残課題)
4. **次タスクへの引き継ぎ** (ブロック解除タスク ID / 派生起票)

### タスク種別ごとのガイドライン

[[03_design/task_completion_criteria|タスク完了基準仕様書]] § 3 を参照:

- 3-1. コード実装タスク (例: F267 / F269 / F141 / F211)
- 3-2. ドキュメント・テンプレタスク (例: F233 / F234 / F271)
- 3-3. データ取得・パイプラインタスク (例: F100 / F101 / F268)
- 3-4. ゲート・承認タスク (例: F266 / F241 / F053)

### レビュー段数

- 重要タスク (Stage 移行ブロッカー / 不可逆操作): Mac mini 完了報告 →
  Claude.ai レビュー → Fujiwara 最終判定
- 通常タスク: Claude 完了報告 → Fujiwara 任意レビュー
- 軽微タスク (工数 < 4 時間 / リスクなし): 自己申告のみ

### アンチパターン (やってはいけない完了報告)

[[03_design/task_completion_criteria|タスク完了基準仕様書]] § 6 を参照。
代表 6 種:

- 「ファイルを作りました」だけ報告 (Wikilink 検証していない)
- 「exit code 0」を成功と見なす (意味的成功条件を確認していない)
- 「将来作るので OK」で先送り (完了基準の矮小化)
- 「動作確認不要なほど自明」(自明性の過信)
- 「テスト PASS」だけで「機能した」と判定 (smoke 未実施)
- 「全項目 PASS」だけで偽 PASS を見逃す (前提不成立の PASS)
