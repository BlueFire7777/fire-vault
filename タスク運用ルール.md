# FIRE 開発タスク運用ルール(v3.3 対応)

本書は、FIRE 統合要件書 v3.3 の開発を Git + Obsidian + .md で運用するためのルール集。

## 1. タスクID採番ルール

**正本: [TODO管理表 Google Sheets](https://docs.google.com/spreadsheets/d/1Zf0rmQHxuBLWsBriV3hKH51eGiJ5u96b/edit?gid=1782363065#gid=1782363065)**

- 全タスクは F001 〜 F999 の連番で Google Sheets の TODO_Master シートに一意採番
- **Google Sheets を Single Source of Truth (SSoT) とする**。ローカルxlsxファイル(FIRE_開発TODO_v8.xlsx)はバックアップ位置づけ
- 新規追加時は末尾に追番
- タスクIDは Git commit、Branch名、PRタイトル、Obsidianノート名すべてで必ず引用する

### 管理の二重化を避ける原則

- ステータス更新 (未着手→実装中→完了) は **Google Sheets でのみ行う**
- Obsidian のタスクノートは「タスクの詳細な実装メモ・ログ」を書く場所
- Obsidian の frontmatter の `status:` は参考情報であり、Google Sheets 側が優先

## 2. Git 運用

### ブランチ戦略

- main - 本番運用ブランチ
- dev - 統合開発ブランチ
- feature/F242-openclaw-setup - タスクID含むfeatureブランチ
- hotfix/FXXX-bug-name - 緊急修正

### コミットメッセージ規約

`<type>(<task_id>): <subject>` の後に本文、最後に Refs: F001, F242 等

type: feat / fix / docs / refactor / test / chore

### FIREリポジトリ個別の Git 設定

StudioZ業務と分離:

```
cd ~/fire
git config user.name "BlueFire7777"
git config user.email "<個人Gmail>"
```

### PRルール

- PRタイトル例: `[F242] OpenClaw セットアップ`
- PR本文にタスクID、完了条件、テスト方法、関連Obsidianノートリンク
- マージ時に Google Sheets TODO_Master のステータスを完了に更新

## 4. 崩してはならない前提(v3.3: 9項目)

1. FIREのエッジは**再現性優位**
2. Pattern Storeは**大量蓄積 + 階層管理**
3. パターンは段階検証を通過して昇格(**Stage飛ばし禁止**: 0→1→2→3)
4. **Evaluation Agentは提案のみ**(自動反映禁止)
5. Mac mini常駐ランタイム前提(**OpenClawで実現**)
6. UIはレトロ炎テーマ
7. 初期本番はデイトレ主軸
8. **(v3.3)全自動は目指さない。Live Advisory(半自動)が FIRE の完成形**
9. 完全放置はしない(監督者付き自律運用)

## 5. 開発着手順(第40章の絶対原則)

1. OpenClaw 基盤が最初(全ての土台)
2. Pattern Store / Feature Store が最優先(エッジの本質)
3. LINE 通知基盤は並行開発可
4. 約定取込は M+2 以降(初期は LINE 手動報告)
5. Stage 飛ばし禁止
