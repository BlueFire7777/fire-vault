### HIGH #1: staging-only DB path guard has a TOCTOU window before write open
- 該当箇所: `scripts/jobs/run_f286_pnl_r3_compute_paper_pnl.py:187`, `scripts/jobs/run_f286_pnl_r3_compute_paper_pnl.py:227`, `scripts/jobs/run_f286_pnl_r3_compute_paper_pnl.py:262`, `scripts/jobs/run_f286_pnl_r3_compute_paper_pnl.py:468`
- 内容: `--write` 時の symlink / resolved basename / hardlink inode 検査は `parse_args()` 内で実行されるが、実際の write open は後段の `_connect_write(args.db_path)` で通常の `sqlite3.connect(str(path))` を使う。read phase と compute phase の後に write phase が来るため、検査済みの `fire.staging.db` がその後 symlink / hardlink / replacement で差し替えられると、検査時点と open 時点の対象が一致する保証がない。
- 影響: 悪意ある、または壊れた並行プロセスが同一 filesystem 上で DB path を差し替えられる前提では、staging-only write guard の保証が弱まる。単独実行の通常運用では発火しにくいが、F282 の「production / develop DB write を構造的に防ぐ」観点では高リスク。
- 推奨修正: write 直前にも `_assert_db_path_not_symlink_attack(args.db_path)` を再実行し、可能なら `_connect_write()` 内で再検査する。より強くするなら、`os.open(..., O_NOFOLLOW)` 相当で file descriptor を取得し、その fd の `st_dev/st_ino` を forbidden DB と比較してから sqlite に渡す設計を検討する。少なくとも read phase 後、write transaction 直前の再 stat は入れるべき。

### MEDIUM #1: output path safety also has a post-validation replacement window
- 該当箇所: `scripts/jobs/run_f286_pnl_r3_compute_paper_pnl.py:136`, `scripts/jobs/run_f286_pnl_r3_compute_paper_pnl.py:348`, `scripts/jobs/run_f286_pnl_r3_compute_paper_pnl.py:356`, `scripts/jobs/run_f286_pnl_r3_compute_paper_pnl.py:528`
- 内容: `--output-json` / `--completion-report` は basename / suffix / existing symlink / hardlink を `parse_args()` で検査しているが、実際の `Path.write_text()` 直前には再検査しない。検査後に出力 path が symlink に差し替えられると、DB/WAL/SHM 破壊防止ガードを迂回され得る。
- 影響: 通常の単独実行では低頻度だが、DB 近傍 path や `/tmp` 配下を使う運用で並行差し替えがあると任意ファイル上書きに近い挙動になる。
- 推奨修正: `_write_outputs()` / `_write_completion_report()` の直前または関数内で `_assert_safe_output_path()` を再実行する。可能なら atomic create (`'x'` mode) または temp file + `os.replace` を使い、既存 symlink を踏まない実装にする。

### MEDIUM #2: decision_label matching is exact and silently excludes whitespace / normalization variants
